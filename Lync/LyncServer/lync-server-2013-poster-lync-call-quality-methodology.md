## <a name="what-is-lync-cqm"></a>什麼是 Lync CQM？

通話品質方法是一種新的疑難排解與服務管理架構，可讓您更好地專注于在 Lync 中改善企業語音服務。 當您使用 CQM 時，需要較少的精力，以確保通話品質與企業語音服務的使用者滿意度。 CQM 在[通話品質方法](http://go.microsoft.com/fwlink/p/?linkid=615208)中會有更完整的說明。 本文和海報都是該內容的摘要。

CQM 會將系統疑難排解分解為三個路徑或 "道路"。 這些是： [伺服器植物] 道路，會在伺服器和連結之間的連結上，[端點道路] 會顯示使用者裝置和用來進行通話的媒體，以及可解決傳統交換電話網絡通話的問題。

每個道路都分為數個與特定區域或主題有關的區段，每個區段定義都會針對可接受的品質等級進行，採取行動來達到該品質等級，並將服務管理方案納入維護該品質等級，然後再移至下一個主題。

海報提供 Lync CQM 做為三台玩家的棋盤遊戲，每個玩家都會透過其中一個道路進行。 下載所含的卡片是用來模擬障礙，以進行必須克服的通話品質。 您可以在三個路徑中包含有關目標及如何取得它們的提示與建議，以及優先順序指導方針，讓您在實際的應用程式中先開始進行，並以平行的方式來處理三個道路。

CQM 在舊版 Lync 中的運作方式為何？ CQM 是 Lync 2013 的新功能，但大部分都可以調整為與 Lync 2010 搭配使用。 CQM 可能會在 Microsoft Office Communicator 上發揮作用，但這是經過測試且不受支援。

如果您有關于如何使用 CQM 的問題，您可以將問題提交至 cqmfeedback@microsoft.com。

## <a name="prioritize-run-trending-queries"></a>優先順序：執行趨勢查詢

CQM 中的第一個步驟是執行兩周的每個趨勢查詢，然後分析結果。 依最大的資料流程參與者、最高不良的資料流程比率，以及受管理的區域（您所控制的）來確定修正動作的優先順序。如果音訊/視頻多重點控制單位（AV MCU）或轉送查詢顯示的結果不佳，請從紅色或伺服器植物道路開始。如果有線或無線查詢顯示的結果不佳，請從藍色或最後一個英里道路開始。如果 VPN 或外部查詢顯示的結果不佳，請從綠色或終點道路開始。

在您選擇開始使用的道路之後，為每一個區域定義一個目標（Assert）、工作以符合該目標（完成），然後執行程式來維持目標（維護）。 您也可以使用這個標牌做為遊戲，瞭解 CQM 背後的原則。

## <a name="pcd"></a>PCD

PreCall 診斷工具（PCD）可協助您找出並診斷周邊網路中的問題（QoE 資料庫不會在您的邊緣或周邊網路上收集資訊），也可讓您在最後一個英里中針對連線進行疑難排解。 此工具既可以是 Windows 8 新式 App，也可在 Windows 桌面應用程式http://apps.microsoft.com/windows/en-us/app/lync-2013-precall-diagnostics/9607fe33-2b51-403d-9615-c23f248e7c88中使用。

## <a name="managedunmanaged"></a>Managed/非託管

Lync Server 部署和網路基礎結構通常可以劃分成受管理且未受管理的空間。 受管理的空間包括您整個內部有線網路和伺服器基礎結構。 未受管理的空間是無線結構和外部網路基礎結構。

如此一來，就能增加資料的明晰性，並協助貴組織將重點放在對使用者的語音及視頻品質有顯著影響的工作負載上。 如果將呼叫放在您所擁有（受管理）與部分其他實體（非託管）控制的基礎結構中，使用者會有不同的品質預期。 這並不表示無線使用者已離開其自己的裝置，以提供絕佳的 Lync 伺服器體驗。

在非託管空間中改善語音品質，需要您在受管理的空間中擁有高品質。 是否認為無線（Wi-fi）是由您的組織決定。 在這兩個空間中，達到良好環境的技巧會有所不同，就像使用這些方案一樣。

## <a name="the-server-plant-road"></a>伺服器植物道路

伺服器工廠中的第1區段會解決 Lync 實現中的實際伺服器。 在實施中收集伺服器本身及其角色的 KHI 資料，並分析結果。 如果有保修措施，請修正發現的任何問題。 有關本主題的詳細資訊，請參閱 KHI 海報隨附的[Lync Server 2013 中的主要健康情況指示](lync-server-2013-poster-key-health-indicators.md)文。

下一個區段是在 AV MCU 伺服器與中繼伺服器之間的媒體資料流程。 首先，決定您的目標是否有不佳的資料流程閥值。 較差的資料流程通常\>是 PacketLossRate .01 \>或 PacketLossRateMax。 另一個預期的目標\<是 PoorStreamsRatio 2%。 接著，使用詳細的查詢來找出含不良資料流程的 AVMCU 和轉送伺服器配對、調查不良資料流程的原因、查看不良資料流程路徑中的網路設備、修正不正確的資料流程，以及為網路定義最佳或「黃金」設定設備. 若要維持您的成就，請執行流程與工具來管理設定的偏移，並報告新的問題區域。

接著，請檢查在中繼伺服器與公用交換電話網絡（PSTN）閘道之間的媒體資料流程。 首先，決定您的目標是否有不佳的資料流程閥值。 較差的資料流程通常\>是 PacketLossRate .01 \>或 PacketLossRateMax。 另一個預期的目標\<是 PoorStreamsRatio 2%。 接著，使用詳細的查詢來找出具有不良資料流程的中繼伺服器和閘道配對、調查不良資料流程的原因、查看不良資料流程路徑中的網路設備、修正不良資料流程，以及為網路定義最佳或「黃金」設定設備. 若要維持您的成就，請執行流程與工具來管理設定的偏移，並報告新的問題區域。

最後，請檢查您 PSTN 閘道的健康情況指標。 找出顯示健康情況的統計資料，並針對它們定義目標。 此處未提供任何特定指導方針，因為有許多可能的閘道可供使用。 建立目標之後，請視需要修正以達到目標;在此程式中，您可能會定義閘道的 "金色" 或最佳配置。 若要維持您的成就，請執行流程與工具來管理設定的偏移，並報告新的問題區域。 請注意，固件和軟體更新可能會改變您的設定，或讓您變更「黃金」設定的定義，所以請小心地進行這些動作。

## <a name="the-last-mile-road"></a>最後一英里的道路

在用戶端連線到網路的兩種方式中，有線預期會提供最高品質，而且必須是您最後一個取得英里問題的初始焦點。 使用 CQM 有線查詢（LastMile\_0\_有線）以及它提供的不良資料流程比率資料。 我們建議您為含\< \> 300 串流的網站定義目標 PoorStreamsRatio 5%）。 若要取得您的目標，請將從最差訂購的子網修正成最佳，並實施 QoS。

在您優化有線連線品質之後，改善無線品質會變得更容易，因為無線基礎結構在每個位置放在有線核心中。 在具有良好有線品質的網站中，較差的無線資料流程，必須有特定的無線元件。 CQM 無線查詢（LastMile\_1\_無線）是在日期範圍執行，並將您環境中的所有內部無線資料流程從 Lync 用戶端傳回或從會議服務器或轉送伺服器返回。 我們建議您為含\< \> 300 串流的網站定義目標 PoorStreamsRatio 5%）。 若要取得您的目標，請將從最差訂購的子網修正成最佳，並實施 QoS。

## <a name="the-end-points-road"></a>端點道路

在與 Lync 搭配使用時，使用耳機和其他已知可產生可接受品質的裝置，在端點中開始查詢。 我們建議使用超過100個\>資料流程之實現的目標 AvgSendListen MOS 3.6。）透過找出有問題的裝置並修正或取代，來取得目標。

接下來，請檢查裝置或電腦處理音訊，以供使用者通話。 建議的目標品質指標是 AudioMicGlitchRate \<= 1。 當您識別使用者系統的最佳系統設定時，請定義包括驅動程式版本在內的 "黃金" 電腦設定。

現在，請檢查音訊資料流程從 Lync 端點系統取得的網路路徑，這可能會導致音訊品質不佳。 如果音訊是穿過 VPN 連線，您可能會看到延遲問題。 如果內部 Lync 用戶端無法為兩方或對等呼叫的另一個內部 Lync 用戶端建立直接媒體資料流程，它會回到可透過 Lync Edge 伺服器中繼的路徑，進而帶來延遲問題，並增加遺失和抖動。 我們建議您將品質指標定義為0% 媒體超過 VPN。 當您修正以達到您設定的目標時，請找出問題的子網，並調查防火牆規則、資料包 shapers，以及其他相關的網路設備設定。

IP 資料包可以使用傳輸控制通訊協定（TCP）或使用者資料包通訊協定（UDP）。 TCP 對於資料流程而言是最佳的。 UDP 是無中斷且更有效率的媒體，因為 TCP 恢復機制無法及時處理即時媒體的遺失。 Lync 一直傾向使用 UDP，但如果無法建立 UDP 會話，就會還原為 TCP。 TCP 上的媒體會話會顯示比 UDP 更差的品質。 我們建議將品質定義為在 TCP 上使用0% 連線。 當您修正以達到您設定的目標時，請找出問題的子網，並調查防火牆規則、資料包 shapers，以及其他相關的網路設備設定。

## <a name="service-management"></a>服務管理

服務管理是 CQM 的結束狀態，以及所有三個道路的目的地。 若要維持高層次的通話品質，請監視以下區域：

1.  **使用者**修正活動應該顯示使用者滿意度中可衡量的增加。 您可以透過問題入場券或其他意見反應機制來衡量這個問題。 您也可以發佈品質統計資料。

2.  **Process** -定義每日、每週及每月處理 operationalize CQM。 [監視律動] 會在您進行修正時以較高的頻率啟動（每日），並隨著您的穩定而移至較低的頻率（每月）。

3.  [**工具**]-識別測量與修正的工具。 您可能會發現自動執行 CQM 查詢以支援您的程式非常有用。 修正可能需要額外的工具，例如，在網路元素上強制執行標準化設定，或在糟糕的資料流程中排除遺失問題。

## <a name="board-game-rules"></a>棋盤遊戲規則

您可以使用這個標牌做為 CQM 的參照，或做為遊戲的做法來練習概念。 若要播放，您需要 1 6 側骰子及隨附的卡片。 可在標準 Avery 5871 名片上列印的卡片可下載版本。

遊戲是針對3個玩家。 玩家可以使用三種路徑來達到所需品質並達到中心服務管理狀態：伺服器工廠、終點及最後一英里。 每個路徑都已停止在您斷言品質目標、實現目標，以及維護系統方面的方式上。 將卡片放在上方所示的區域中，然後繪製5張卡片。 查看您繪製的卡片，並將它們放在相關的版面區段上。 每個玩家都會逐步透過其路線中的卡片逐步移動，以斷言品質目標、取得這些目標，以及維持服務等級。 當所有玩家都達到中心服務管理狀態時，遊戲就會完成。 遊戲卡下載提供更詳細的規則。

若要**斷言**品質目標，請查看適用于該目標的參數，並向外移出您要的內容，而不會選擇接受。 我們建議使用起點，但您必須進行最終通話。 例外狀況是 KHI 資料，而您應該使用 Microsoft 建立的標準。 請參閱隨附的 KHI 海報。

若要在遊戲中**實現**，請使用提供的卡片來代替 KHI 資料和系統查詢。 如果您在遊戲開始時沒有繪製與指定方位有關的卡片，您可以繼續進行。 如果有相關的卡片，請滾動骰子。 如果您是在卡片上所顯示的數位底下滾動，表示您已成功完成。 如果您要將指標放在指定的數位上，您必須從投影片組中繪製另一個卡片。 如果卡片指出兩個或多個玩家需要滾片，就必須全部順利完成。

若要在遊戲中**維護**，請將與 Lync 環境相關的服務管理方案向外醒目。
