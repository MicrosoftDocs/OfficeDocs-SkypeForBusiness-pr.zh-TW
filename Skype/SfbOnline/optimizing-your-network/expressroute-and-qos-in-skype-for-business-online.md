# <a name="expressroute-and-qos-in-skype-for-business-online"></a>商務用 Skype Online 中的 ExpressRoute 與 QoS

透過使用 Azure ExpressRoute for Office 365 和商務用 Skype Online 的私人網路絡連線，連線到 Office 365。 您的商務用 Skype 應用程式的專用連線將可提供可靠且可預測的效能，以及從公用網際網路以外的隱私權。 您現在可以在 Office 365 和商務用 Skype Online 中購買更好的網路連線，新增可預測性、商業課程可靠性，並隨附正常運作時間 SLA。
  
> [!NOTE]
> 新版本的頻寬計算機可供使用：[商務用 Skype、頻寬計算機](https://go.microsoft.com/fwlink/?LinkId=715766)。 不過，這份檔中的指示會使用 Lync 2010 和2013頻寬計算機。 
  
## <a name="skype-for-business-online-and-expressroute"></a>商務用 Skype Online 與 ExpressRoute

使用 Microsoft 的 ExpressRoute 合作夥伴，您可以連線多種 Office 365 應用程式，包括在雲端的專用連線中的商務用 Skype Online。 不過，商務用 Skype 的即時語音及視頻通訊功能，必須具備專門設定以支援這些 Office 365 即時工作負載的網路服務。 這包括具備足夠頻寬的網路，以傳送所需的流量，並且能夠支援服務品質（QoS）來為您的使用者供應商務類別體驗。
  
本檔的設計目的是協助您、管理員和網路設計者瞭解支援即時通訊所需的特殊挑戰，以及 Microsoft 提供的工具，以協助您設計能支援這些功能的網路需求，並逐步引導您使用案例研究設計程式。 
  
本檔的第一個部分會逐步引導您完成案例研究，以協助您使用[Lync 2010 和2013頻寬計算機](https://go.microsoft.com/fwlink/?LinkId=690282)來評估大型、多網站商務用 Skype For business ExpressRoute 部署的網路需求。 本檔的第二部分為您提供服務品質（QoS）的基本概念，深入瞭解支援商務用 Skype 即時通訊的特定技術詳細資料，以及所需的特定類型的網路服務需要.
  
這裡的所有資訊都可為您提供 QoS 與 ExpressRoute 的技術詳細資料與瞭解，瞭解您所面臨的特定挑戰，並提供可讓您成功使用的工具和技術的相關知識。在您的商務用 Skype 網路上部署 ExpressRoute。

### <a name="setting-up-the-network"></a>設定網路

若要為 Dewey 定律 LLC 提供一致且優質的即時服務，必須符合幾個基本需求：
  
- 他們想要在電源發生故障期間提供語音服務，因此其網路發佈交換器和路由器必須在上提供 power 乙太網路（PoE） IEEE 802.3 或802.3。
    
- 網路交換器和路由器也必須使用不斷電供應系統（UPS），讓它們在電源發生故障期間仍能繼續運作。
    
    他們擁有與其局域網辦公室的 Wi-fi 連線，因此我們強烈建議他們使用[商務用 skype](https://go.microsoft.com/fwlink/?LinkId=690281)中的商務用 skype wi-fi 基礎結構合作夥伴。
    
    > [!TIP]
    >  建議使用 802.11 n 和 802.11 ac 無線存取點。
  
- 而且最重要的是，所有辦公室中的所有局域網網路都必須設定為提供服務品質（QoS）。 這包括電腦、膝上型電腦和任何網路硬體（例如交換器和路由器）。
    
現在您已掌握基本功能，可為 Dewey 定律 LLC 供應商業成績的語音服務。我們建議您使用從網路服務合作夥伴連線至 Azure ExpressRoute 服務的多重通訊協定標籤（MPLS） IP 服務。 MPLS 提供 IP 服務，並針對延遲、抖動和資料包遺失進行效能保證。 不過，如果 MPLS 無法使用，您也可以使用乙太網路連接到其中一個 ExpressRoute 資料 exchange 合作夥伴。
  
MPLS 提供者提供數種服務層級，但每個類別都使用不同的術語來加以識別。 您必須與提供者密切合作，確保他們瞭解您在[Lync 2010 和2013頻寬計算機](https://go.microsoft.com/fwlink/?LinkID=690282)中輸入的資料，以及適用于不同 Office 365 即時工作負載的選項程式.
  
商務用 Skype 應用程式的資料可以對應到適當的 MPLS 服務類別，有兩個選項可供選擇：
  
- 使用 DiffServ 控制點（DSCP）標記流量的端點
    
- 以網路存取控制清單（ACL）為基礎
    
若要實現端點標記，您必須針對 Dewey 定律 LLC 設定所有加入網域的 Windows 電腦。 若要使用適當的 DiffServ 控制點（DSCP）標示每個資料包，然後在所有 office 網站上的所有網路交換器和路由器上執行 QoS，以確保 QoS 標記已維護且不會移除。 網路資料包上的 DSCP 標記會告訴服務提供者該網路資料包的優先順序。 **第2部分中的 [QoS] 區段中有更多關於 DSCP 的資訊。**
  
針對以網路 ACL 為基礎的作業，DSCP 優先順序標記是在上游路由器上實現，且是以 UDP 來源埠為基礎。 每個應用程式的建議埠範圍會列于[Lync Server 的 [網路規劃]、[監視] 和 [疑難排解](https://go.microsoft.com/fwlink/?LinkId=690286)] 區段2.6.1.1 中。 您可以將這項功能與 Dewey 定律 LLC 的整體 QoS 實施及設計，以及資料包標記不一致的可能性進行協調，這一點非常重要。
  
每個 ExpressRoute 網路服務提供者都會有一個適用于即時語音和影片的服務類別（QoS）。 這個 COS 稱為「加急轉寄」（EF），用於語音和「有保證的轉寄」（AF）。 您必須小心地調整您針對語音 EF 流量購買的頻寬數量。 原因是，當您傳送的語音流量超過您提供的服務類別時，語音服務的語音類別非常 unforgiving。
  
> [!TIP]
>  在服務語音類別上傳送的任何流量超過服務提供者的承諾，都會立即捨棄，這會直接影響語音品質。
  
看看 Dewey 定律的整體設計 LLC。 您必須準確判斷他們所需的網路頻寬，以支援其網路的語音流量，並以語音（亦即 DSCP EF 46）標示每個語音資料包（且僅限語音資料包）。
  
若要在商業網路中實現 QoS，端點或路由器必須以適當的第3層優先順序指示器（亦即 DSCP）標示每個資料包。 沿著整個網路路徑，每個切換和路由器都必須開啟 QoS 選項。 只要有一個網路交換器或沒有開啟 QoS 的路由器，就可以將語音或視頻資料包上的 QoS 標記從該交換器或路由器上剝離。 這會有效地停用所有下游交換器和路由器中的 QoS，這會減少擁有 ExpressRoute 的值。
  
這也要求在每個點上定義第3層和第2層 QoS 優先順序的關聯。 第2層優先順序機制是在有線網路和 802.11 e/WMM for Wi-fi 網路的 IEEE 802.1 p 中定義。 更重要的是，網路服務提供者的 MPLS 網路所面向的網路路由器必須在所有出站資料包上保留 DSCP 設定，才能維持適當的 MPLS 服務類別。 
  
> [!TIP]
>  如需有關 QoS 設定的特定詳細資料，請參閱 2.6[的網路規劃、監視及疑難排解（Lync Server]( https://go.microsoft.com/fwlink/?LinkId=760669)）一節。 您也可以參閱[規劃商務用 Skype 2015 的網路需求](https://go.microsoft.com/fwlink/?LinkId=690287)，以取得更多的網路規劃需求。
  
### <a name="ordering-network-access-services"></a>排序網路存取服務

當您具備 QoS 網路的先決條件和機制以支援 ExpressRoute 之後，下一步就是為 ExpressRoute 網路存取服務下訂單。 從 Microsoft 網路服務提供者合作夥伴針對 Dewey 定律 LLC 排序 ExpressRoute access services 時，您必須提供兩個專案：
  
- 將每個網站連接至 ExpressRoute 和 Office 365 所需的頻寬總量。
    
- 支援在 Dewey 定律 LLC 中使用之商務用 Skype app 所需的每個服務類別所需的總頻寬。 服務類別的頻寬需求是由您預計來自各種商務用 Skype 應用程式（例如語音、影片、IM、目前狀態及螢幕共用）所預期的流量所驅動。
    
### <a name="determining-bandwidth-requirements-for-skype-for-business-applications"></a>判斷商務用 Skype 應用程式的頻寬需求

針對 Dewey 定律 LLC。如果您已確定所需的總頻寬，您現在必須知道如何在各種不同的服務類別之間劃分總頻寬量。 例如，每個商務用 Skype 應用程式的頻寬是多少。
  
若要在每個 Dewey 定律 LLC 上判斷這些需求。 網站，您將會使用[Lync 2010 和2013頻寬計算機](https://go.microsoft.com/fwlink/?LinkID=690282)。 此計算機是一種 Excel 工具，可讓您指定各種商務用 Skype 應用程式的預期用途，包括語音、影片、會議和螢幕共用。 計算機會自動為其網路上的每個網站產生頻寬和 CoS 需求的估計。 當您下載 Lync 2010 和2013頻寬計算機時，系統也會下載使用者指南，讓您瞭解如何使用它。 
  
為了協助您使用試算表，試算表中的各種儲存格都以色彩標示：
  
- **綠色**這些是一般資料輸入區域。
    
- **黃色**這些是先進的資料輸入區域。 您可以變更這些動作，但請謹慎進行。
    
- **紅色**這些是唯讀區域，且是已鎖定的輸入值且無法變更。
    
- **灰色**這些只是顯示區域。 它們是來自一般輸入區域的結果或資料。
    
Dewey 定律 LLC 的設計程式。 首先，請將使用者的使用者描述成不同的「角色」。 針對您定義的每個角色，您可以指定其預期使用各種商務用 Skype 應用程式（"無"、"低"、"中型"、"High" 或三個已定義的 [自訂] 設定）。 這些選取專案會在「Persona」工作表中找到。 提供每個選項（"低"、"中" 或 "High"）的特定用法，但每個選項的預設值都可以變更。 透過識別每個網站的每個角色的使用者數目，計算機就可以計算每個位置所需的總頻寬。
  
您也可以指定使用的音訊和視頻編解碼器、是否使用轉寄錯誤修正，以及會影響頻寬需求的其他系統參數。 您可以使用 Lync 2010 和2013頻寬計算機中的預設設定，或選取不同的編解碼器及其他系統參數。 針對 Dewey 定律 LLC 的網站設計，您可以使用預設設定。 不過，若要從任何預設設定中變更，都有一個下拉式功能表，其中包含所有可用的選項。 每個選項所使用的頻寬都包含在「編解碼器」工作表中。 當您變更任何設定時，會更新在每個網站上的頻寬和服務類別（CoS）組合的變更。 具備此功能可讓您測試不同的可能配置，並查看變更對其頻寬需求有何影響。

### <a name="the-real-time-problem"></a>即時問題

供應商務品質的語音和影片服務會針對 IP 網路施加特殊需求。 即時流量使用使用使用者資料包協定（UDP）傳送的即時傳輸通訊協定（RTP）。 與針對錯誤的傳輸控制通訊協定（TCP）進行編號與測試，以及包含偵測及重新傳輸遺失或 errored 訊息的其他機制，UDP 不提供這種類型的可靠性。 如果郵件由於錯誤而損毀，或由於緩衝區溢位而遺失，則會遺失。 已選擇與 RTP 搭配使用 UDP，因為即時流量的性質是，即使遺失的郵件沒有重發，也會有太晚的時間來對語音訊息流程產生任何積極的影響。
  
瞭解遺失語音資料包的影響之後，設計人員就可以使用兩種方法來改善 IP 語音及視頻的效能：
  
- 當資料包遺失時，讓語音編碼/解碼變得更具彈性。 您可以使用轉寄錯誤修正（FEC）來修正在 Office 365 即時傳輸中發現的問題百分比，或設計語音解碼系統來嘗試遮罩遺失之資料包的效果。是 Microsoft 編解碼器的特性。 
    
- 使用的傳輸服務會使用品質服務機制，以保證網路的效能與中斷、資料包遺失、抖動，以及資料包間延遲的變化。
    
復原式語音編碼只會解決資料包遺失的問題，因此使用網路來攜帶即時語音和影片的網路必須具備最小化延遲和抖動的機制。 即便是復原的編碼，如果遺失太多的資料包，接收站就沒有足夠的資訊來重新構造可識別的語音信號版本。 可能導致語音品質明顯下降之資料包所占的百分比，視使用的語音編碼技術而有所不同。 不過，在任何情況下，後續資料包的遺失字串都有很大的問題。
  
最小化延遲很重要，因為過度延遲可能會影響交談流程，並為喇叭帶來令人討厭的干擾。 最佳做法請告訴我們聲音的端對端延遲（我們稱為「嘴對耳」延遲）必須保持在150毫秒（毫秒）以下。 單向，不是「往返行程」的延遲。 當然，延遲會隨著傳播延遲或傳送信號在電纜上實際旅行時所需的時間，增加較長的傳輸連結（例如穿過 oceans 的傳輸）。
  
當延遲超過150毫秒時。 一種方式是，它在喇叭上具有奇怪的效果。 Psychologically，喇叭的大腦中的時鐘熄滅，讓他們認為收件者沒有聽到對方的聲音，而且他們會重複他們所說的最後一件事。 這會與來自最遠端的延遲回應相衝突。 如果您曾在衛星通道上說話，您會認識到此效果。 在附屬通道上，單向延遲大約是250毫秒。，這遠遠超過了允許的延遲。

### <a name="basic-concepts-of-quality-of-service-qosclass-of-service-cos"></a>服務品質的基本概念（QoS）/Class （CoS）

在 IP 中，服務品質（QoS）描述了任何用來針對一些其他資料包提供優先順序處理的機制。 根據國際電訊同盟（ITU）定義，QoS 會包含連接的所有品質內容，包括延遲、遺失、信號到雜訊比率、crosstalk、迴響、中斷、頻率回應、loudness 層級等。 我們在資料包網路中所指的 QoS 是更正確的方法（CoS），側重于改善延遲、抖動與資料包遺失的效能，但我們會繼續使用 QoS 詞彙，就像更常使用的字詞。
  
在 IP 網路通話中提供 QoS，以進行兩個主要元件：
  
- 在每個連結上為即時流量預留已定義的頻寬量;如果即時流量不需要此頻寬，您可以在其他流量中使用它。 一般指導方針是應該為語音流量指派任何連結容量的30% 以上。
    
- 在標頭中標示具有優先順序指示器的資料包，告知路徑中的開關和路由器應該指派的資料包優先順序。
    
在交換器或路由器收到資料包時，會將它移到下一個腿或跳躍的輸出佇列中。 不同的優先順序層級會有不同的輸出佇列。 交換器或路由器使用的演算法比較低優先順序的佇列更頻繁地服務高優先順序佇列。
  
難題是在第2層（亦即乙太網或 Wi-fi 層）和第3層（亦即 IP 圖層）中實現不同的 QoS 技術。 這些不同的 QoS 執行可能必須在網路的每個交換器和路由器中進行設定，以及您的網路與網路服務提供者網路之間的介面。
  
有兩種選項可讓您將各種商務用 Skype 應用程式的資料對應到適當的服務類別：
  
- 使用區別服務控制點（DSCP）標記流量的終點 
    
- 網路存取控制清單（ACL）

### <a name="layer-2-qos-ieee-8021pwi-fi-multi-media-ieee-80211e"></a>第2層 QoS： IEEE 802.1 p/Wi-fi 多媒體（IEEE 802.11 e）

雖然 DSCP 是在第3層實現 QoS 的標準機制，但無線（亦即乙太）和無線（亦即 Wi-fi 網路）的第2層 QoS 機制會有所不同。 有線網路的 QoS 機制是在 IEEE 802.1 p 標準中定義;WLAN QoS 機制是在 IEEE 802.11 e 中定義，Wi-fi 聯盟識別為 "Wi-fi 多個媒體已驗證" （WMM 驗證）。
  
IEEE 802.1 p 使用3位優先順序碼位（PCP）來識別訊息的優先順序;PCP 是乙太網上報頭中的32位欄位的一部分，也會攜帶 VLAN 識別碼。 PCP 值的定義如下所示。

在其中，IEEE 802.1 p 的實現方式與 DSCP 基本相同，其通信量會針對每個優先順序等級排序為不同優先順序佇列，但 Wlan 的共用媒體性質是針對不同的方法。 雖然存取點和用戶端會針對不同的優先順序層次維持不同的輸出佇列，但在無線電通道上傳送畫面的方式也有所不同。
  
在 Wi-fi 網路中，所有與存取點相關聯的用戶端都共用一個半雙工通道（亦即只有一個用戶端站或存取點可以一次傳送）。 若要將無線電通道上發生碰撞的可能性降至最低，請在傳送畫面前等待頻道在已定義的一段時間（即「畫面間間距」）中，如果當站傳送時頻道忙，則會在隨機時間 period. 傳送框架後，如果寄件者沒有收到來自收件者的確認訊息，就會假設發生衝突或其他失敗，並在嘗試存取要傳送的無線電通道之前傳回隨機間隔。 後置時間間隔是隨機的，以降低相同兩個電臺會再次發生的機率。
  
為了區分無線通道的存取權，IEEE 802.11 e/WMM 會針對不同的流量類別，定義稱為「仲裁內幀間隔」（AFIS）和不同後端範圍的不同預傳輸等待間隔;定義了四個稱為「存取類別」的優先順序等級。
  
您可以將較短的 AFIS 值指派給較高優先順序的幀來提供優先順序。 因此，如果一個工作站正在等待傳送語音畫面，而另一個站正在等待傳送資料框架，語音畫面就會先傳送給您。 從技術角度來看，語音及影片畫面會獲指派相同的 AFIS 值，但影片畫面的後退間隔範圍較高。 因此，語音畫面和影片框架在第一次嘗試發生衝突時，語音畫面總是會儘早重新傳輸。 IEEE 802.1 p 與 IEEE 802.11 e e 之間的關聯如下所示：

### <a name="implementing-qos-using-network-access-control-list-acl"></a>使用網路存取控制清單（ACL）來實現 QoS

在 ExpressRoute 配置中實現 QoS 的替代方法是使用網路存取控制清單（ACL）。 在該方法中，不是讓端點在每個資料包的標頭中插入適當的 DSCP 標記，而是根據 UDP 來源埠，以上游路由器來完成標記。 所有的交換器和路由器都必須設定為支援 QoS，以確保已維護 DSCP 設定。 更重要的是，連線至服務提供者網路的路由器必須在每個資料包的標頭中保留 DSCP，因為該 DSCP 設定實質是您向網路服務提供者指示該資料包應該如何處理。
  
每個商務用 Skype 應用程式的建議埠範圍，都列在 [Lync Server 指南] 的 [[網路規劃]、[監視] 和 [疑難排解](https://go.microsoft.com/fwlink/?LinkId=690286)] 區段2.6.1.1 中。 我們很重要的一點，就是要與組織的整個 QoS 原則進行協調，而且您應該針對不同的 QoS 原則以及可能的資料包 remarking 不一致的 lookout。
  
雖然使用的主要原因是 QoS 與 MPLS 網路服務，以確保即時語音與影片的使用者體驗，這些相同的功能也可以套用到資料應用程式。 MPLS 網路不是同等地對待所有的應用程式，而是可以讓組織將某些資料應用程式的優先順序提供給其他人。 使用 MPLS，您可以將信用卡交易或螢幕共用等即時應用程式的優先順序設定為較少的時間，例如電子郵件。
  
### <a name="understanding-the-types-of-ip-network-services--basic-ip-and-mpls"></a>瞭解 IP 網路服務的類型-基本 IP 與 MPLS

以「最佳做法」為原則運作的原始 IP 資料包轉寄。 這表示將這些 IP 資料包轉寄的路由器可以盡最大的目的來傳送給他們的目的地，但在這些情況下或是否會到達目的地時絕對沒有任何保證。 這就是如何使用基本的網際網路服務，包括您的家用網際網路連線、今天的工作。 其思想是，如果特定應用程式需要可靠性，就會在通訊協定堆疊的較高層級提供此功能。 可靠的傳遞機制是傳輸控制通訊協定（TCP）。 [使用者資料包協定] （UDP）是用來進行即時語音和影片的，是不可靠（亦就是「最大努力」）傳遞機制。 
  
多協定標籤切換（MPLS）是針對網路服務提供者所開發的一種方式，提供可保證延遲、抖動及資料包遺失的 IP 服務。 為提供這些效能保證，MPLS 會從傳統 IP 中取得一些 unpredictability。 首先，不是讓每個資料包找到其目的地的路由器對路由器的方式，而是將「虛擬線路」連線中的所有資料包都路由到目的地，而 MPLS 會將所有的資料包傳送給固定路由稱為標籤切換路徑（LSP）。 如果路徑中的其中一個連結失敗，所有使用該連結的 Lsp 都會快速重新路由。
  
當資料包傳送到 MPLS 網路時，網路服務提供者的 edge 路由器會將額外的標頭附加至資料包，該資料包包含用來將它轉寄到適當 LSP 的標籤。 標籤會由 MPLS 網路另一端的邊緣路由器來去除。
  
在簡化轉寄程式的旁邊，MPLS 的另一個優點是，網路管理系統會知道要在網路上的每個連結上傳輸哪些連線。 透過控制流量透過網路路由的方式，操作員可以保證每個路徑都會提供的 QoS。 因此，不像傳統或基本 IP 的最佳努力效能，MPLS 運算子可以提供具有可預測效能的 IP 服務。 該 LSP 也能讓 MPLS 在與傳統的網際網路服務更安全的情況中執行。 因此，使用基本 IP 服務，我們可以希望網路能夠正常運作，以提供優質的語音及使用 FEC 及更具彈性的語音編碼來改善可能性，但透過使用 MPLS，我們可以確定。
  
MPLS 提供者提供數種服務梯度類別，很遺憾會使用不同的字詞來識別它們。 您必須與提供者密切合作，以確保他們瞭解[Lync 2010 和2013頻寬計算機](https://go.microsoft.com/fwlink/?LinkID=690282)的產出，以及不同 Office 365 即時工作負載應用程式的建議選項。
