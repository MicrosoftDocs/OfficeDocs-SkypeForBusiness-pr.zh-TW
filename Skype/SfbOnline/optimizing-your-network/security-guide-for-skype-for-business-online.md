# <a name="security-and-skype-for-business-online"></a>安全性與商務用 Skype線上

[!INCLUDE [sfbo-retirement](../../Hub/includes/sfbo-retirement.md)]

商務用 Skype線上 (SfBO) 是 Microsoft 365 和 Office 365 服務的一部分，遵循所有安全性最佳做法和程式，例如透過深度防護、服務中的客戶控制措施、安全性強化及營運最佳做法等服務層級安全性。 如需詳細資料，請參閱 Microsoft 信任中心 https://microsoft.com/trustcenter) (。

## <a name="trustworthy-by-design"></a>從設計上值得信賴
商務用 SkypeOnline 是按照 SDL (的 Microsoft 信任計算安全性開發週期) 設計及開發，方法如下 https://www.microsoft.com/sdl/default.aspx 所述。 建立更安全整合通訊系統的第一步是設計威脅模型，並在設計每一項功能時同步測試。 程式碼撰寫和慣例已內建許多安全性相關的改進。 在程式碼移轉至最終產品之前，建置階段工具會偵測緩衝區溢位和其他潛在的安全性威脅。 當然，設計時不可能預防到所有未知的安全性威脅。 沒有任何系統可以保證絕對的安全性。 不過，由於產品開發從一開始即採用安全的設計原則，商務用 Skype Online 將產業標準安全性技術納入其架構的基本部分。 

## <a name="trustworthy-by-default"></a>透過預設而值得信賴
根據預設，商務用 Skype Online 中的網路通訊會加密。 要求所有伺服器都使用憑證，並使用由由所有伺服器使用、TLS、Secure Real-Time 傳輸通訊協定 (SRTP) ，以及其他業界標準加密技術 ，包括 256 位進步加密標準 (AES) 加密，所有 商務用 Skype Online 資料會受到保護。

## <a name="how-sfbo-handles-common-security-threats"></a>SfBO 如何處理一般安全性威脅
本節指出 SfBO 服務安全性的常見威脅，以及 Microsoft 如何減輕每個威脅。

### <a name="compromised-key-attack"></a>盜取金鑰攻擊
金鑰是用來加密、解密或驗證機密資訊的密碼或數位。 在公用金鑰基礎結構 (PKI) 中，有兩個機密金鑰必須經過考慮：每個憑證擁有者都有的私密金鑰，以及通訊合作夥伴成功進行識別和工作階段金鑰交換後所使用的工作階段金鑰。 當攻擊者決定私密金鑰或工作階段金鑰時，會發生入侵金鑰攻擊。 當攻擊者成功判斷金鑰時，攻擊者可以使用金鑰解密加密的資料，而不需要寄件者知道。

商務用 SkypeOnline 會使用 Windows Server 作業系統中的 PKI 功能，以保護傳輸層安全性與 TLS (加密) 資料。 用於媒體加密的金鑰會透過 TLS 連線來交換。 

### <a name="network-denial-of-service-attack"></a>網路拒絕服務攻擊
當攻擊者妨礙有效使用者正常使用網路並使網路失去功能時，就表示發生了拒絕服務攻擊。 藉由使用拒絕服務攻擊，攻擊者可以：
- 對受攻擊網路上所執行的應用程式和服務傳送無效資料，以干擾其正常功能。
- 傳送大量流量，讓系統超載直到其停止回應或只能緩慢回應合法要求。
- 隱藏攻擊證據。
- 妨礙使用者存取網路資源。

SfBO 可執行 Azure DDOS 網路保護，並節流來自相同端點、子網和聯盟實體的用戶端要求，以防範這些攻擊。

### <a name="eavesdropping"></a>竊聽
當攻擊者獲得網路中資料路徑的存取權，並有能力監視及讀取流量時，就可能發生竊聽。 這種攻擊也稱為探查或窺探。 如果流量是純文字，攻擊者就能在獲得路徑的存取權時讀取流量。 藉由控制資料路徑上的路由器來執行攻擊便是其中一例。 

SfBO 使用相互的 TLS (MTLS) ，用於 Microsoft 365 或 Office 365 中的伺服器通訊，以及從用戶端到服務的 TLS，使此攻擊在受到攻擊的時段內難以達成。 TLS 會對各方實施驗證，並加密所有流量。 這雖然不會妨礙竊聽，但除非加密遭到破解，否則攻擊者將無法讀取流量。

TURN 通訊協定可用於即時媒體用途。 TURN 通訊協定不會要求流量必須加密，而是會透過郵件完整性來保護所傳送的資訊。 此通訊協定雖對竊聽不設防，但其所傳送的資訊 (也就是 IP 位址和連接埠) 只要藉由查看封包的來源和目的地位址，就能直接擷取到。 SfBO 服務會使用從幾個專案衍生的金鑰檢查郵件的訊息完整性，以確保資料有效，包括永遠不會以純文字送出的 TURN 密碼。 SRTP 則可用於媒體流量，而且也會進行加密。

### <a name="identity-spoofing-ip-address-spoofing"></a>身分識別詐騙 (IP 位址詐騙)
當攻擊者確定網路、電腦或網路元件的 IP 位址並未經授權就使用時，就表示發生詐騙。 如果攻擊成功，攻擊者就能假裝自己是經過該 IP 位址正常識別的實體。 在 Microsoft Lync Server 2010 中，只有在系統管理員同時執行下列兩項操作時，才能執行此情況：
- 已配置的僅支援傳輸控制通訊協定 (TCP)  (不建議使用，因為 TCP 通訊未加密) 。
- 將這些連接的 IP 位址標示為信任的主機。 

由於 TLS 會驗證所有 (並加密所有) ，因此傳輸層安全性與 TLS) 較沒有關系。 使用 TLS 可防止攻擊者對特定連線 (例如相互 TLS 連線) 執行 IP 位址詐騙。 但攻擊者仍可能詐騙 SfBO 使用的 DNS 伺服器位址。 不過，由於 SfBO 中的驗證是使用憑證執行，因此攻擊者不需要有效的憑證來詐騙通訊中的其中一方。

### <a name="man-in-the-middle-attack"></a>中間人攻擊
當攻擊者在不知道兩個通訊使用者的情況下，透過攻擊者的電腦重新路由兩個使用者之間的通訊時，即會發生中間攻擊。 攻擊者可在流量傳送給預期的收件者之前，監視並讀取流量。 通訊中的每個使用者都會在不知情的情況下傳送流量給攻擊者，以及接收來自攻擊者的流量，卻又認為他們在與預期的使用者通訊。 如果攻擊者能修改 Active Directory Domain Services 而將其伺服器新增為信任的伺服器，或修改網域名稱系統 (DNS) 而讓用戶端透過攻擊者連線到伺服器，就會發生這種情況。 

此外，兩個用戶端之間的媒體流量也會發生一場中間攻擊，但 SfBO 點對點音訊、視像和應用程式共用流會使用 SRTP 加密，使用在 TLS 上使用會話初始通訊協定 (SIP) 進行對等程式之間的加密金鑰。 

### <a name="rtp-replay-attack"></a>RTP 重播攻擊
當兩方之間的有效媒體傳輸遭到惡意攔截並重新傳輸時，就會發生重播攻擊。 SfBO 會使用 SRTP 與安全訊號通訊協定，讓接收者能夠維護已接收的 RTP 封包索引，並比較每個新封包與索引中已經列出的封包，以保護傳輸不受重播攻擊。

### <a name="spim"></a>Spim
Spim 是未經要求的商業立即訊息或目前狀態訂閱要求。 Spim 本身雖然不會危害網路，但卻很煩人，且會降低資源可用性和生產力，並可能導致網路遭到危害。 例如，使用者傳送要求來對彼此進行 Spim 攻擊。 使用者雖可彼此封鎖以避免這種攻擊，但如果在同盟時發生了協調一致的 Spim 攻擊，則除非您停用合作夥伴的同盟，否則將難以解決這個問題。

### <a name="viruses-and-worms"></a>病毒與蠕蟲
病毒是一個程式碼單位，其目的是重現其他類似的程式碼單位。 若要發生作用，病毒需要有宿主，例如檔案、電子郵件或程式。 和病毒一樣，蠕蟲也是一種程式碼單位，其編碼目的是要產生其他類似的程式碼單位，但不像病毒一樣需要宿主。 病毒和蠕蟲主要會在用戶端之間的檔案傳輸過程中，或在其他使用者傳送 URL 時現身。 舉例來說，如果電腦上有病毒，病毒就可以使用您的身分識別，並以您的名義傳送立即訊息。 標準用戶端安全性最佳做法 (例如定期掃描病毒) 可緩解此問題。 

## <a name="personally-identifiable-information"></a>個人識別資訊

- **增強的目前狀態資料** &nbsp; &nbsp; &nbsp;增強的目前狀態資料是使用者可以選擇共用或不共用至聯盟合作夥伴或與組織內部連絡人的連結的資訊。 此資料不會與公用 IM 網路的使用者共用。 用戶端策略和其他用戶端組組可能會與系統管理員進行一些控制。 在 SfBO 服務中，您可以為個別使用者配置增強的目前狀態隱私權模式，以防止不在使用者連絡人清單上的 SfBO 使用者看到使用者的目前狀態資訊。

## <a name="security-framework-for-sfbo"></a>SfBO 安全性架構
本節提供構成 Microsoft SfBO 安全性架構的基本元素概觀。 這些元素如下所示：
- Azure Active Directory (AAD) 提供使用者帳戶的單一信任後端存放庫。 
- PKI (PKI) 會使用信任的憑證授權單位 (CA) 驗證服務器，並確保資料完整性。
- 傳輸層安全性 (TLS) HTTPS (HTTPS) ，以及相互 TLS (MTLS) 啟用端點驗證和 IM 加密。 系統會使用安全即時傳輸通訊協定 (SRTP) 加密點對點音訊、視訊和應用程式共用串流，並進行完整性檢查。
- 盡可能提供使用者驗證的產業標準通訊協定。

本節中的主題說明這些基本元素如何運作，以增強 SfBO 服務的安全性。
 
### <a name="azure-active-directory"></a>Azure Active Directory
Azure Active Directory 會作為 Microsoft 365 和 Office 365 的目錄服務。 其儲存了所有使用者目錄資訊和原則指派。 

### <a name="public-key-infrastructure-for-sfbo"></a>SfBO 的公用金鑰基礎結構
SfBO 服務仰賴憑證進行伺服器驗證，並建立用戶端與伺服器之間以及不同伺服器角色之間的信任鏈。 PKI Windows 伺服器公用金鑰基礎結構 (PKI) 提供基礎結構，以建立及驗證此信任鏈。
憑證是數位識別碼。 它們根據名稱識別伺服器，並指定其屬性。 若要確保憑證上的資訊有效，憑證必須由憑證授權單位 (CA) 由用戶端或連接到伺服器的其他伺服器信任。 如果伺服器僅與私人網路上的其他用戶端和伺服器連接，則 CA 可以是企業 CA。 如果伺服器與私人網路外部的實體互動，可能需要公用 CA。

即使憑證上的資訊有效，也必須有一些方法可以驗證呈現憑證的伺服器實際是憑證所代表的伺服器。 這是 PKI Windows的地方。
每個憑證都連結到一個公用金鑰。 在憑證上命名的伺服器會保留只有它知道對應的私密金鑰。 連接用戶端或伺服器會使用公用金鑰加密隨機的資訊，並將它傳送至伺服器。 如果伺服器解密資訊並以純文字方式退貨，則連接實體可以確定伺服器保留憑證的私密金鑰，因此是憑證上指定的伺服器。

#### <a name="crl-distribution-points"></a>CRL 發佈點
SfBO 要求所有伺服器憑證都包含一或多個憑證撤銷清單 (CRL) 發佈點。 您可以從 CRL 發佈點 (CDP) 來下載 CRL，以便驗證憑證自發行以來尚未遭到撤銷，且憑證仍在有效期限內。 CRL 發佈點是安全的 HTTP，會以 URL 的形式註明在憑證的屬性中。 SfBO 服務會使用每個憑證驗證檢查 CRL。

#### <a name="enhanced-key-usage"></a>增強金鑰使用方法
SfBO 服務的所有元件都需要所有伺服器憑證，以支援增強型金鑰使用量 (EKU) 以用於伺服器驗證。 為伺服器驗證設定 EKU 欄位表示該憑證可供用來驗證服務器。 此 EKU 對 MTLS 非常重要。 

### <a name="tls-and-mtls-for-sfbo"></a>SfBO 的 TLS 和 MTLS
TLS 和 MTLS 通訊協定可在網際網路上提供加密通訊和端點驗證。 SfBO 會使用這兩種通訊協定來建立信任的伺服器網路，並確保該網路的所有通訊都經過加密。 伺服器之間的所有 SIP 通訊都發生在 MTLS 上。 從用戶端到伺服器的 SIP 通訊會以 TLS 進行。

TLS 可讓使用者透過他們的用戶端軟體，驗證他們連結的 SfBO 伺服器。 在 TLS 連線中，用戶端會要求伺服器提供有效憑證。 若要有效，憑證必須是由同時受到用戶端信任的 CA 所發行，且伺服器的 DNS 名稱必須與憑證上的 DNS 名稱相符。 如果憑證有效，用戶端就會使用憑證中的公開金鑰來加密要用於通訊的對稱加密金鑰，而只讓憑證的原始擁有者可以使用其私密金鑰來解密通訊內容。 這時所產生的連線會受到信任，且自此之後就不會遭到其他受信任伺服器或用戶端查問。 在此範圍內，與 Web 服務 (安全通訊端層) SSL 層可以關聯為 TLS 型。

伺服器對伺服器的連線會仰賴相互 TLS (MTLS) 來進行相互驗證。 在 MTLS 連線中，產生訊息的伺服器和接收訊息的伺服器會交換雙方都信任的 CA 所核發的憑證。 憑證可讓伺服器雙方證明彼此的身分識別。 在 SfBO 服務中，會遵循此程式。

TLS 和 MTLS 可同時防範竊聽和中間人攻擊。 在中層攻擊中，攻擊者在不知道任何一方的情況下，透過攻擊者的電腦重新路由兩個網路實體之間的通訊。 TLS 和 SfBO 的受信任伺服器的規格，會使用使用兩個端點之間使用公用金鑰加密方式協調的端對端加密，降低應用程式層上發生人員中間攻擊的風險，而攻擊者必須擁有具有對應私密金鑰的有效且信任的憑證，併發行至用戶端所通訊的服務名稱，以解密通訊。

SfBO 使用 TLS 和 MTLS 加密立即訊息。 伺服器對伺服器的流量無論是局限於內部網路還是跨越內部網路周邊，都需要使用 MTLS。

#### <a name="media-encryption"></a>媒體加密
媒體流量會使用安全 RTP (SRTP) 來加密，SRTP 是即時傳輸通訊協定 (RTP) 的設定檔，可為 RTP 流量提供機密性、驗證和重播攻擊防護。 SRTP 會使用透過安全亂數產生器所產生，並透過訊號 TLS 通道來進行交換的工作階段金鑰。 此外，在中繼伺服器及其內部下一個躍點之間往兩個方向流動的媒體也會使用 SRTP 加密。 

SfBO 會產生使用者名稱/密碼，以在 TURN 上安全地存取媒體轉場。 媒體轉傳會以 TLS 安全 SIP 通道交換使用者名稱/密碼。

#### <a name="fips"></a>FIPS
SfBO 使用 FIPS (聯邦資訊處理標準) 加密金鑰交換的相容演算法。 

### <a name="user-and-client-authentication"></a>使用者和用戶端驗證 
信任的使用者是認證已由 AAD 在 Microsoft 365 或 Office 365。 

驗證是將使用者認證佈建到受信任伺服器或服務的程序。 SfBO 會根據使用者的狀態和位置，使用下列驗證通訊協定。
- **新式驗證** 是 MICROSOFT 對用戶端與伺服器通訊的一種 2.0 的實現。 它啟用安全性功能，例如憑證式驗證、多重要素驗證和條件式存取。 若要使用 MA，就必須為線上租用戶和用戶端啟用 MA。 根據預設，在 2017 年 5 月之後建立 SfBO 租使用者已啟用 MA。 對於在此時間之前建立之租使用者，請遵循此處的指示將其開啟。 下列用戶端都支援 MA：商務用 Skype 2015 或 2016 用戶端、mac 商務用 Skype、Lync 2013 用戶端、3PIP IP 電話、iOS 和 Android。 
- **組織識別碼** 用於未啟用新式驗證 (或無法) 。
- **針對所謂的** 匿名使用者摘要通訊協定。 匿名使用者是外部使用者，他們尚未識別 Active Directory 認證，但受邀參加內部部署會議，並擁有有效的會議金鑰。 摘要驗證不會用於其他用戶端互動。

SfBO 驗證由兩個階段組成：
1. 用戶端與伺服器之間會建立安全性關聯。
2. 用戶端和伺服器會使用現有的安全性關聯來簽署他們傳送的郵件，並驗證他們收到的郵件。 在伺服器上啟用驗證時，不會接受來自用戶端的未驗證郵件。

使用者信任會附加到來自使用者的每封郵件，而不是使用者身分識別本身。 伺服器會檢查每封郵件是否具有有效的使用者認證。 如果使用者認證有效，則郵件不受第一個伺服器以及 SfBO 中所有其他伺服器之挑戰。

擁有由聯盟合作夥伴所發出有效認證的使用者受到信任，但選擇性地受到其他限制限制所禁止，無法享有給予內部使用者的完整許可權範圍。

針對媒體驗證，ICE 和 TURN 通訊協定也會使用如 IETF TURN RFC 所述的摘要查問。 有關詳細資料，請參閱 [媒體橫穿](#external-user-av-traffic-traversal)。

用戶端憑證提供替代方式，讓使用者通過 SfBO 驗證。 使用者沒有提供使用者名稱和密碼，而是擁有對應到憑證的憑證和私密金鑰，這是解決加密問題所需的。 

### <a name="windows-powershell-and-sfbo-management-tools"></a>Windows PowerShell和 SfBO 管理工具
在 SfBO 中，IT 系統管理員可以透過 O365 系統管理入口網站，或是使用租使用者遠端 PowerShell (TRPS) 。 租用戶系統管理員會使用新式驗證來向 TRPS 驗證。

### <a name="configuring-access-to-sfbo-at-your-internet-boundary"></a>在網際網路邊界上配置 SfBO 的存取權
若要讓 SfBO 正常運作 (能夠加入會議等 ) 的使用者，客戶必須設定其網際網路存取權，以便允許 SfBO 雲端服務外發 UDP 和 TCP 流量。 有關詳細資料，請參閱這裡： https://support.office.com/article/Office-365-URLs-and-IP-address-ranges-8548a211-3fe7-47cb-abb1-355ea5aa88a2#bkmk_lyo 

### <a name="udp-3478-3481-and-tcp-443"></a>UDP 3478-3481 和 TCP 443

用戶端會使用 UDP 3478-3481 和 TCP 443 埠向 A/V Edge 服務要求服務。 用戶端會使用這兩個埠來分別配置 UDP 和 TCP 埠，讓遠端方能夠連接。 若要存取 A/V Edge 服務，用戶端必須先與 SfBO 註冊機構建立經過驗證的 SIP 訊號會話，才能取得 A/V Edge 服務驗證認證。 這些值會在整個受 TLS 保護的訊號通道之間傳輸，而且是由電腦產生，以防範字典攻擊。 然後，用戶端可以使用這些認證來與 A/V Edge 服務一起摘要驗證，以配置埠以用於媒體會話。 初始配置要求會從用戶端寄來，並回復來自 A/V Edge 服務的 401 非ce/挑戰訊息。 用戶端會傳送第二個配置，其中包含使用者名稱和雜湊郵件驗證碼 (HMAC) 使用者名稱和非名稱的雜湊值。 

此外，也提供序號機制，以防止重播攻擊。 伺服器會根據使用者名稱和密碼本身的知識來計算預期的 HMAC，如果 HMAC 值相符，則執行配置程式。否則，封包會被刪除。 此相同的 HMAC 機制也會用於此通話會話中的後續郵件。 此使用者名稱/密碼值的生命週期上限為八小時，用戶端在此時會為後續通話重新取得新的使用者名稱/密碼。

### <a name="udptcp-5000059999"></a>UDP/TCP 50，000-59，999
TCP 50，000 外發適用于 SfBO，包括應用程式和桌面共用、檔案傳輸。 UDP/TCP 50，000-59，999 埠範圍適用于需要 A/V Edge 服務 NAT/防火牆橫向服務的 Microsoft Office Communications Server 2007 合作夥伴的媒體會話。 由於 A/V Edge 服務是使用這些埠的唯一程式，因此埠範圍的大小不會指出潛在的攻擊面。 良好的安全性作法是，不進行不必要的網路服務，一直將聆聽埠總數降到最低。 如果網路服務未執行，遠端攻擊者無法利用這項服務，並且降低主機電腦的攻擊面。 不過，在單一服務中，減少埠數量並不提供相同的優點。 A/V Edge 服務軟體已開啟 10，000 個埠，因此不再受到攻擊。 此範圍內的埠配置會隨機完成，目前未配置的埠不會聆聽封包。

### <a name="external-user-av-traffic-traversal"></a>外部使用者 A/V 流量橫排
若要讓外部使用者和內部使用者交換媒體，需要 Access Edge 服務來處理設定和關閉會話所需的 SIP 訊號。 它也需要 A/V Edge 服務做為媒體傳輸的轉場。 下列圖說明通話順序。


![會議加入中的通話順序](media/sfbo-call-sequence-security.png) 

1. 使用者會收到一封電子郵件，其中包含加入 SfBO 會議的邀請。 電子郵件包含會議金鑰和以 HTTP 為基礎的 URL 連結至會議。 金鑰和 URL 對於特定會議都是唯一的。<p>使用者會按一下電子郵件中的會議 URL 以啟動使用者電腦上用戶端偵測程式，以啟動加入程式。 如果偵測到用戶端，此用戶端會啟動。 如果未偵測到，使用者會重新導向至 Web 用戶端。<p/>
2. SfBO 用戶端會傳送包含使用者認證之 SIP INVITE。 聯盟或遠端使用者會使用企業認證加入會議。 針對聯盟使用者，SIP INVITE 會先送到其主伺服器，驗證使用者，然後轉寄 INVITE 至 SfBO。 需要匿名使用者才能通過摘要驗證。<p>SfBO 會驗證遠端或匿名使用者，並通知用戶端。 如步驟 2 所述，加入會議的聯盟使用者會由他們的企業進行驗證。<p/>

3. 用戶端會傳送 INFO 要求，將使用者新加入 A/V 會議。

    A/V 會議會傳送包含要呈現給 A/V 會議 Edge 服務之權杖的 Add User 回應，以及其他資訊。

    [附注] 所有前述 SIP 流量都經過 Access Edge 服務。

    用戶端會連接到 A/V 會議伺服器，此伺服器會驗證權杖，並代理要求 ，其中包含另一個授權權杖，至內部 A/V 會議伺服器。 A/V 會議伺服器會驗證原本通過 SIP 通道發行的授權權杖，進一步確保有效的使用者正在加入會議。

4. 在用戶端和 A/V 會議伺服器之間，媒體連接會經過 SRTP 的交涉及設定。
5. 使用者會收到一封電子郵件，其中包含加入 SfBO 會議的邀請。 電子郵件包含會議金鑰和以 HTTP 為基礎的 URL 連結至會議。 金鑰和 URL 對於特定會議都是唯一的。

## <a name="addressing-threats-to-sfbo-conferences"></a>解決 SfBO 會議的威脅

SfBO 提供企業使用者建立和加入即時 Web 會議的功能。 Enterprise使用者也可以邀請沒有 AAD、Microsoft 365或Office 365帳戶的外部使用者參與這些會議。 由具有安全且經過驗證身分識別的聯盟合作夥伴雇用的使用者也可以加入會議，如果升級的話，也可以擔任簡報者。 匿名使用者無法以簡報者的身分建立或加入會議，但可在加入會議後受到提升而成為簡報者。

讓外部使用者參與 SfBO 會議會大幅提升此功能的價值，但也帶來一些安全性風險。 為了防範這些風險，SfBO 提供下列額外防護措施：
- 參與者角色會決定會議控制許可權。
- 參與者類型，可讓您限制特定會議的存取權。
- 已定義的會議類型會決定哪些類型的參與者可以出席。
- 會議排程僅限於擁有 AAD 帳戶和 SfBO 授權的使用者。
- 匿名，即未經驗證的使用者，想要加入電話撥入式會議撥打其中一個會議存取號碼，然後系統會提示他們輸入會議 ID。 系統也會提示未經驗證的匿名使用者記錄其名稱。 錄製的名稱會識別會議未驗證的使用者。 在至少一位領導者或經過驗證的使用者加入會議之前，匿名使用者不得加入會議，而且無法指派預先定義的角色。

### <a name="participant-roles"></a>參與者角色
會議參與者分成三個群組，每個群組都有各自的權限和限制：
-  &nbsp; 召集人 &nbsp;建立會議的使用者，無論是即席會議或排程。 召集人必須是經過驗證的企業使用者，且能控制會議的所有使用者層面。
- **簡報者** &nbsp; &nbsp;有權在會議上展示資訊的使用者，無論支援何種媒體。 根據定義，會議召集人同時也是簡報者，可決定還有誰可以成為簡報者。 召集人可在排定會議時或於會議進行當下做出這項決定。
- **出席者** &nbsp; &nbsp;受邀參加會議但無權擔任簡報者的使用者。

簡報者也可以在會議進行期間將出席者提升為簡報者角色。

### <a name="participant-types"></a>參與者類型

會議參與者也會依位置和認證來分類。 您可以使用這兩種特性來指定哪些使用者可以存取特定會議。 使用者大致可以分成下列類別：
1.  **屬於租使用者的使用者** &nbsp; &nbsp;這些使用者擁有租使用者Azure Active Directory的認證。<br/>
    a. *在公司網路* 內部 – 這些使用者從公司網路內加入。<br/>b. *遠端使用者*：這些使用者是從公司網路外部加入會議的。 這些人包括在家、在外工作的員工和其他人 (例如受信任供應商的員工)，且必須已獲得可在服務期間使用的企業認證。 遠端使用者可以建立和加入會議，並擔任簡報者。
2.  **不屬於租使用者的使用者** &nbsp; &nbsp;這些使用者在租使用者Azure Active Directory沒有認證。<br/>a. *聯合使用者* - 聯合使用者擁有與聯盟合作夥伴的有效認證，因此會視為由 SfBO 驗證。 已加入會議後，聯盟使用者可以加入會議，並升級為簡報者，但他們無法在與它們建立聯盟的企業中建立會議。<br/>b. *匿名使用者*：匿名使用者沒有 Active Directory 身分識別，且未與租用戶同盟。 

客戶資料顯示，許多會議都涉及外部使用者。 這些相同的客戶也想要在允許這些使用者加入會議之前，先確認外部使用者身分識別。 如下一節所述，SfBO 會限制會議存取權給明確允許的使用者類型，而且要求所有使用者輸入會議時，都須提供適當的認證。

### <a name="participant-admittance"></a>參與者准入
在 SfBO 中，匿名使用者會移轉到稱為大廳的等候區域。 簡報者可以准許這些使用者加入會議或拒絕他們。 這些使用者會移轉到大廳，主席會收到通知，然後使用者等候，直到領導者接受或拒絕他們或他們的連線時間過長。在大廳中，使用者會聽到音樂。 

根據預設，從 PSTN 撥入的參與者會直接前往會議，但此選項可以變更為強制撥入參與者前往大廳。  
會議召集人可以控制參與者是否可加入會議而不必在大廳中等候。 每一場會議都可以透過下列任一方法來設定為啟用存取：
- **僅自己，會議召集人** &nbsp; &nbsp;除了召集人之外，所有人都必須在大廳等候，直到獲准。
- **我從公司邀請的人** &nbsp; &nbsp;您公司的任何人都可以直接加入會議，即使未受邀也一樣。
- **我組織的任何人** &nbsp; &nbsp;租使用者或租Microsoft 365 Office 365中的所有 SfBO 使用者都可以加入會議，而不需要在大廳等候，即使不在通訊群組清單上的使用者也一樣。 所有其他使用者 ，包括所有外部和匿名使用者，都必須在大廳等候，直到獲准。
-  &nbsp; 任何人 &nbsp;任何 (沒有限制) 存取會議連結的人可直接進入會議。
當指定只有 (已鎖定) 以外的任何方法時，會議召集人也可以指定以電話撥入的人，而忽略大廳。 

### <a name="presenter-capabilities"></a>簡報者的能力
會議召集人可控制參與者是否可以在會議期間進行簡報。 每一場會議都可以進行設定，而將簡報者限制在下列任何人：
- **僅召集人** &nbsp; &nbsp;只有會議召集人可以出席。
- **公司人員** &nbsp; &nbsp;所有內部使用者可以進行展示。
- **包括公司外部人員在內的所有人** &nbsp; &nbsp;所有 (沒有限制) 加入會議的人可以出席。
- **我選擇的人** &nbsp; &nbsp;會議召集人會指定哪些使用者可以加入簡報者清單來展示。
