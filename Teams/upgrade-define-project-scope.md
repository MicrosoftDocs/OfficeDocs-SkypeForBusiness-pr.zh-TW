# <a name="define-your-project-scope"></a>定義專案範圍

![升級歷程圖表，醒目提示專案定義階段](media/upgrade-banner-project-definition.png "升級歷程階段，重點放在專案定義階段")

本文是您在升級歷程的專案定義階段的一部分，在您從已確定的專案關係人中建立贊助聯盟和專案小組之後，您所完成的活動就是您專案成功的關鍵。 繼續之前，請確認您已完成下列活動：

- [已登記專案的干係人](upgrade-enlist-stakeholders.md)

您需要花時間來定義您的專案構想、範圍、目標及管轄，以確保所有專案干係人都能在相同的最終結果中對齊和運作。 此功能特別重要，因為技術就緒小組與使用者準備小組會獨立運作，以將各自的棋子放在一起。 完成本節之後，請在整個專案中參考它，以確保您已順利完成您想要的結束狀態。 使用以下所述的目標來評估您的結果，並視需要減少。

## <a name="project-vision"></a>專案構想

您的願景是「大圖片」或最終狀態，回答問題，「為什麼我們要執行這份專案？」 理想的設想是解決貴組織的商務驅動因素與使用者值-新增角度，如下列範例所示：

- **組織商務版驅動程式**：在 Microsoft 團隊中進行標準化與我們的數位工作區轉換，讓我們能夠提高運作效率、消除重複的解決方案，並節省 USD5 萬美元的功能。
- **使用者值-新增**： Microsoft 團隊 (1) 可為專案筆記、Office 檔、小組成員、交談和會議提供單一位置，以節省時間。 (2) 透過使用集中式連絡人清單和持續性聊天追蹤進行通訊，以快速存取您的交談，以及 (3) 減少嘗試找出遺失的電子郵件附件的不滿，因為它在一個位置儲存和存取檔案。

請考慮下列討論點，以協助精簡您的願景：

- 目前業務程式的描述

- 現有業務程式面臨的挑戰

- 技術可以如何協助克服這些難題

- 若要解決這些難題，預期和可衡量的商業結果

> [!TIP]
> 找出使用案例和角色，進一步細化您的專案構想。

## <a name="project-scope"></a>專案範圍

您的願景可能只會在一段時間內透過各種階段來實現。 專案範圍會定義您專案的重點， _並讓您的專案小組_ 專注于目前的工作，讓您能夠實現您的長期設想。 例如，您的範圍可能會要求您執行試驗、部署特定的工作負載（例如語音或會議），或在進行商務用 Skype 時，按照您規劃的方式，在一段時間內啟用團隊升級。 做為專案範圍的一部分，您應該評估：

- [各種共存模式](https://aka.ms/SkypeToTeams-Coexist)，對於您的組織而言是最佳的。
- 在您移至團隊之前，要讓商務用 Skype 與團隊共存的最佳方式。
- 您是否應該執行 [試驗](https://aka.ms/SkypeToTeams-Pilot) 以驗證貴組織中的技術與使用者準備情況。

## <a name="project-goals"></a>專案目標

您的目標會定義您想要的結果，並讓您衡量專案的成功率。 目標可以定義為 [ _目標] 和 [主要結果_ ] (OKRs) ，而專案成功的度量可以定義為 _金鑰成功指示器_ (KSIs) 。 您必須在定義 OKRs 和 KSIs 中完整地參與專案的干係人，以協助確保他們感覺擁有權，並將這些成功的量度與已定義的專案工作對齊。 目標應該包含技術與使用者焦點的成功。

- **OKRs** 包含您在專案開始時所設定的目標，以及您在已定義節奏上測量的主要結果 (例如 [每月] 或 [每季季度]) 。 透過查看您的主要結果，您可以確保專案的交付專案排程正常，或是找出並減少問題，讓您的專案回到正軌中。OKRs 通常會分類為「已達到」或「未取得」。
- **KSIs** 測量主要結果的品質與成功率，並以良好和/或不正確的結果來補充 OKRs 的二進位性質。 在定義 KSIs 時，建議您使用「特定、可測量、可賦值、切實可行、時間相關的「 (智慧型) 準則：
  - 特定：針對特定區域進行改進
  - 可測量：量化或至少建議的指示、進度
  - 可賦值：指定要執行的工作
  - 真實：指出可實際達到哪些結果、給定可用的資源
  - 時間相關：指定可以何時達到結果

## <a name="risks-and-mitigation"></a>風險與緩解措施

 有了任何專案，可能會發生無法預料的事件或其他因素，並將您的專案扔掉追蹤。您必須積極評估潛在的風險，並定義緩解方案，以解決可能發生的問題，讓您的專案可以繼續向您的目標。 _風險註冊_ 是一個絕佳的工具，可讓您追蹤專案風險，以及他們可能會造成的影響，以及捕獲您的緩解方案。 下表顯示範例風險寄存器。

## <a name="timeline"></a>時間表

隨著您升級歷程的範圍，請務必設定主要里程碑的時程表 (例如，除了完成日期之外，還可以為所有使用者啟用小組，並) 所有使用者。 已定義的時程表可協助您的專案小組磁片磁碟機保持一致的結束狀態，並通知適當的工作返回排程，協助確保您的專案保持在正軌。請考慮不太快的時程表， (可能會忽視任務) 或太遠 (可能會) 遺失勢頭。 理想的時程表帳戶：

- **針對合規性與使用者案例需求的產品準備**：請參閱 [產品藍圖](https://aka.ms/O365Roadmap) ，以估量團隊將何時為貴組織做好準備。
- **升級群組**：判斷您是否要透過升級群組來啟用團隊或升級使用者，這可能會影響整體升級歷程的時程表。
- **組織因素，例如變更凍結、會計年度結束、部署週期**：討論並考慮任何可能會影響升級時程表的內部程式。
- **在同一時間或周圍進行的其他變更**：考慮將變更或間距捆綁起來，以協助良好的使用者體驗，並將任何對生產力的影響降至最低。
- 資源 **：請** 與您的專案干係人確認資源分派，以確保您加入的專案小組有足夠的頻寬來完成所有必要的工作。

參考點是提供 [升級 Pro 歷程](https://aka.ms/UpgradePro)之升級前、升級和升級後階段的範例時程表，我們鼓勵您根據需要調整，以符合貴組織的特定需求。

完成上述所述的活動之後，您就應該為您的專案打下堅實的基礎。 繼續進行您的 [技術準備](https://aka.ms/SkypeToTeams-TechnicalReadiness) 及 [組織就緒](https://aka.ms/SkypeToTeams-UserReadiness) 計畫活動。

請記住：2021年7月31日之後，商務用 Skype Online 將會停用。 在這段時間之後，商務用 Skype Online 服務將無法再存取或受支援。 若要最大限度地實現效益並確保貴組織有適當的時間來完成升級，我們鼓勵您立即開始進行您的 Microsoft 團隊歷程。
