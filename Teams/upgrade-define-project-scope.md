## <a name="project-goals"></a>Project目標

您的目標會定義您想要的結果，並能夠衡量專案的成功。 目標可定義為目標與關鍵結果 _(_ OKRs) ，而專案成功的度量可以定義為 KSIs (關鍵) 。  您必須讓專案專案關係人全面參與定義 OKRs 和 KSIs，協助確保他們具有擁有權感，並調整這些成功量值與已定義專案工作。 目標應包含技術與以使用者為焦點的成功組合。

- **OKRs** 包含您于專案開始時設定的目標，以及您根據已定義的步 (例如每月或每季) 。 您可以檢查重要結果，確保您的專案交付專案已排程，或找出並減輕問題，讓專案回到正軌。OKRs 通常分類為「已達」或「未達」。
- **KSIs** 會測量關鍵結果的品質和成功，並詳述良好和/或不佳的結果，以補充 OKRs 的二進位性質。 定義 KSIs 時，建議您在 SMART 準則中，使用「特定、可測量、可指派、實際、 (時間) 準則：
  - 特定：針對特定領域進行改進
  - 可測量：量化或至少建議進度的指示器
  - 可指派：指定誰將執行此工作
  - 實事求是：說明在提供可用資源時，可以實際取得哪些結果
  - 時間相關：指定何時可以取得結果

## <a name="timeline"></a>時程表

當您設定升級旅程的範圍時，請務必設定重要里程碑的時程表 (例如，除了完成日期之外Teams為所有使用者商務用 Skype) 啟用) 。 已定義的時程表可協助您的專案小組朝一致的結束狀態方向努力，並告知正確的回程排程，協助確保您的專案保持進度。考慮一個時程表，當任務可能 (時，可能會忽略) 或太 (，而動力可能會遺失) 。 以下為理想的時程表帳戶：

- **合規性和使用者** 案例需求的產品準備狀態：請參閱產品藍圖 [](https://aka.ms/O365Roadmap)，以判斷Teams何時可供貴組織使用。
- **升級群組**：決定是否要啟用Teams升級群組來升級使用者，這可能會影響整體升級歷程的時程表。
- **組織因素，例如變更凍結、會計年度** 結束、部署週期：討論及考慮可能會影響升級時程表的任何內部程式。
- **同時或** 大約同時發生的其他變更：請考慮將變更進行並聯或將變更間距縮小，以利於良好的使用者體驗，並盡可能降低對生產力的任何影響。
- **資源：** 與專案專案關係人確認資源配置，以確保您彙集的專案小組有足夠的頻寬來完成所有必要的工作。
