# <a name="dimensions-and-measurements-available-in-call-quality-dashboard-cqd"></a>[通話品質儀表板] (CQD) 中提供的維度和度量

Microsoft Teams 和 商務用 Skype Online 的通話品質儀表板 (CQD) 可讓您更加瞭解使用這些服務撥打的通話品質。 本主題提供有關透過 CQD 顯示的維度和度量單位的詳細資訊。 若要深入瞭解 CQD，請參閱 [使用 CQD 管理 Microsoft Teams 中的通話和會議品質](quality-of-experience-review-guide.md)。

## <a name="first-and-second-endpoint-classification"></a>第一個和第二個端點分類

CQD 中許多維度和度量單位都標示為第一個或第二個。 下列邏輯決定串流或通話中包含哪些端點會標示為第一個：

- 當有伺服器參與串流或通話時， (AV MCU、中接伺服器等) 的伺服器端點會視為 「第一」。
- 除非串流位於兩個伺服器端點之間，否則用戶端端點會被視為 Second。
- 如果兩個端點都屬於相同類型，則第一個與第二個端點是根據使用者代理程式類別的內部排序來設定，以確保訂單一致。

例如，每一列都代表一對包含在串流中的使用者代理程式：

|來電者的使用者代理程式類別 |來電者的使用者代理程式類別 |第一個端點 |第二個端點|首先是來電者
|:--- |:--- |:--- |:--- |:--- |
|AV-MCU |OC (商務用 Skype 用戶端)  |AV-MCU |OC (商務用 Skype 用戶端)  |真 |
|OC (商務用 Skype 用戶端)  |AV-MCU |AV-MCU |OC (商務用 Skype 用戶端)  |假 |
|OC (商務用 Skype 用戶端)  |OC (商務用 Skype 用戶端)  |OC (商務用 Skype 用戶端)  |OC (商務用 Skype 用戶端)  |假 |
|AV-MCU |中繼伺服器 |中繼伺服器 |AV-MCU ||
|中繼伺服器 |AV-MCU |中繼伺服器 |AV-MCU |真 |
|OC (商務用 Skype 用戶端)  |OC Phone (商務用 Skype IP Phone)  |OC (商務用 Skype 用戶端)  |OC Phone (商務用 Skype IP Phone)  |真 |
|OC Phone (商務用 Skype IP Phone)  |OC (商務用 Skype 用戶端)  |OC (商務用 Skype 用戶端)  |OC Phone (商務用 Skype IP Phone)  |假 |

> [!NOTE]
> 第一個和第二個分類與來電者或來電者是分隔的。 First Is Caller 維度可用來協助識別來電者或來電者是哪一個端點。

## <a name="dimensions"></a>尺寸

維度資訊的一部分是以上傳到 CQD 入口網站的資料為基礎。 許多維度值也可以做為篩選。

### <a name="dimension-data-types-and-units"></a>維度資料類型和單位

#### <a name="boolean"></a>Boolean

布林值一律為 True 或 False。 在某些情況下，True 也可以表示為 1，而 False 可以表示為 0。

#### <a name="range"></a>範圍

以值範圍或陣列形式提供的維度會使用下列格式顯示：

 _\<sort order string\> [\<lower bound inclusive\> - \<upper bound exclusive\>_

例如，Duration (Minutes) 維度代表以秒為單位的通話持續時間，值會回報為值範圍。

|持續時間 (分鐘)  |如何解譯 |
|:--- |:--- |
|062： [0 - 0)  |串流持續時間 = 0 分鐘 |
|064： [1 - 2)  |1 分鐘< = 串流持續時間 < 2 分鐘 |
|065： [2 - 3)  |2 分鐘 < = 串流持續時間 < 3 分鐘 |
|066： [3-4)  |3 分鐘 < = 串流持續時間 < 4 分鐘 |
|  | |

用來 \<sort order string> 控制呈現資料時的排序次序，並可用於篩選。 例如，[工期] (分鐘) < 「065」 上的篩選會顯示持續時間小於 2 分鐘的串流 (篩選器必須使用前置字元 '0'，才能如預期般) 。 排序次序字串的實際值沒有意義。

> [!NOTE]
> 您可能會發現特定維度的範圍似乎無效。 例如，當 100 是 Wifi 訊號強度的最大值時，會顯示 082 的通話訊號強度：[100 - 110) 範圍。 這是因為在 CQD 資料模型中將數位指派給範圍的方式所導致。 如果整數值是 99，則會將它計入 081：[90 - 100) 範圍。 如果該值為 100，則會將它計入 082：[100 - 110) 範圍。 這不會指出有大於 100% 的 Wifi 訊號強度值。

#### <a name="enumeration-strings"></a>列舉字串

CQD 使用的字串通常衍生自資料檔案，而且在允許的長度內幾乎可以是任何字元組合。 某些維度看起來像字串，但因為它們只能是預先定義值的簡短清單之一，因此它們是列舉，而非真正的字串。 某些列舉字串也會以成對方式使用。

#### <a name="enumeration-pair"></a>列舉配對

做為列舉配對提供的維度會使用下列格式顯示：

 _\<enumeration value from one end point\> : \<enumeration value from the other endpoint\>_

列舉值的順序是一致的，但不會反映第一個或第二個端點的順序。

例如，[網路連線詳細資料配對] 會顯示兩個端點的 [網路連線詳細資料] 值：

|網路連線詳細資料配對 |如何解譯 |
|:--- |:--- |
|有線：有線 |第一個和第二個端點都使用有線乙太網路連線。 |
|有線 ：wifi |第一個端點使用的是有線乙太網路連線，第二個端點是用於連線Wi-Fi，第二個端點則是使用有線乙太網路連線，第一個端點則是使用Wi-Fi連線。 |
|：無線 |第一個端點使用 WiFi 連線，而第二個端點使用的網路連線為未知，或第二個端點使用 WiFi 連線，而第一個端點使用的網路連線為未知。 |
| | |

下表列出維度可能空白的可能原因。 如果 QoE 記錄可用維度為 False，許多維度和度量都會是空白的。 這通常會在通話未成功建立，或用戶端無法傳送遙測至服務時發生。

### <a name="notes-on-measurements"></a>度量的附注

#### <a name="accuracy-limitations"></a>精確度限制
某些使用者和通話計數度量取決於對資料集執行明顯的計數作業來計算計數。 根據執行作業的列數而定，相異計數運算目前會有高達 0.2% 的固有錯誤。 若要獲得最精確的磁片區，您應該使用串流計數量詞，因為它們不依賴這個明顯的計數運算。 篩選以降低資料量可能會減少錯誤，但無法在個別通話和使用者計數中消除此錯誤來源。 您可以在這裡找到有關此限制的詳細資訊 [dcount 匯總函數](/azure/data-explorer/kusto/query/dcount-aggfunction)。
