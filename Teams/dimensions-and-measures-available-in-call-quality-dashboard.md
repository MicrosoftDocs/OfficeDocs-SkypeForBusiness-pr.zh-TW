# <a name="dimensions-and-measurements-available-in-call-quality-dashboard-cqd"></a>通話品質儀表板和 CQD (中可用的維度) 

電話品質儀表板 (CQD) Microsoft Teams 商務用 Skype可讓您更瞭解使用這些服務撥打的通話品質。 本主題提供有關透過 CQD 顯示之維度和度量的詳細資訊。 若要深入瞭解 CQD，請參閱使用[CQD](quality-of-experience-review-guide.md)在 Microsoft Teams。

## <a name="first-and-second-endpoint-classification"></a>第一個和第二個端點分類

CQD 中的許多維度和度量單位會標示為第一個或第二個。 下列邏輯會決定串流或通話中涉及的端點會標示為第一個：

- 當伺服器 (串流或通話時) AV 微控制器、中繼伺服器等的 Server 端點會視為第一個端點。
- 除非串流位於兩個 Server 端點之間，否則用戶端端點會視為第二個。
- 如果兩個端點類型相同，則根據使用者代理程式類別的內部順序來設定第一個和第二個端點，以確保順序一致。

例如，這裡的每一列代表串流中涉及的一組使用者代理程式：

|來電者的使用者代理類別 |受叫者的使用者代理人類別 |第一個端點 |第二個端點|第一個是來電者
|:--- |:--- |:--- |:--- |:--- |
|AV-微控制器 |OC (商務用 Skype用戶端)  |AV-微控制器 |OC (商務用 Skype用戶端)  |真 |
|OC (商務用 Skype用戶端)  |AV-微控制器 |AV-微控制器 |OC (商務用 Skype用戶端)  |假 |
|OC (商務用 Skype用戶端)  |OC (商務用 Skype用戶端)  |OC (商務用 Skype用戶端)  |OC (商務用 Skype用戶端)  |假 |
|AV-微控制器 |中繼伺服器 |中繼伺服器 |AV-微控制器 ||
|中繼伺服器 |AV-微控制器 |中繼伺服器 |AV-微控制器 |真 |
|OC (商務用 Skype用戶端)  |OC 電話 (商務用 Skype IP 電話)  |OC (商務用 Skype用戶端)  |OC 電話 (商務用 Skype IP 電話)  |真 |
|OC 電話 (商務用 Skype IP 電話)  |OC (商務用 Skype用戶端)  |OC (商務用 Skype用戶端)  |OC 電話 (商務用 Skype IP 電話)  |假 |

> [!NOTE]
> 第一個和第二個分類與呼叫者或來電者端點是分開的。 第一個呼叫者維度可用來協助識別哪個端點是來電者或來電者。

## <a name="dimensions"></a>尺寸

維度資訊部分取決於上傳到 CQD 入口網站的資料。 許多維度值也可以做為篩選。

### <a name="dimension-data-types-and-units"></a>維度資料類型和單位

#### <a name="boolean"></a>Boolean

布林值一直是 True 或 False。 在某些情況下，True 也可以表示為 1，而 False 可以表示為 0。

#### <a name="range"></a>範圍

提供為範圍或值群組的維度會使用下列格式顯示：

 _\<sort order string\> [\<lower bound inclusive\> - \<upper bound exclusive\>_

例如， (分鐘) 維度代表通話持續時間，以碼錶示，而值會以值範圍報告。

|持續時間 (分鐘)  |如何解譯 |
|:--- |:--- |
|062：[0 - 0)  |串流持續時間 = 0 分鐘 |
|064：[1 - 2)  |1 分鐘< = < 2 分鐘 |
|065：[2 - 3)  |2 分鐘< = < 3 分鐘 |
|066：[3 到 4)  |3 分鐘< = < 4 分鐘 |
|  | |

用來 \<sort order string> 在呈現資料時控制排序次序，並可用於篩選。 例如，在持續時間 (分鐘) < "065" 上的篩選會顯示持續時間小於 2 分鐘的資料流程 (因此篩選需要前導的 "0" 才能如預期) 。 排序次序字串的實際值沒有意義。

> [!NOTE]
> 您可能會注意到，在給定維度中似乎不正確範圍。 例如，082 顯示通話的 WiFi 訊號強度：[100 - 110) 範圍當 100 為 WiFi 訊號強度的最大值時。 這是因為數位在 CQD 資料模型中如何指派給範圍。 如果一個數值是 99，它會計入 081：[90 - 100) 範圍。 如果該值為 100，它會計入 082：[100 - 110) 範圍。 這不表示已報告超過 100% 的 WiFi 訊號強度值。

#### <a name="enumeration-strings"></a>列舉字串

CQD 所使用的字串通常是從資料檔案衍生，而且這些字串可以是允許長度內的字元組合。 某些維度看起來像字串，但因為它們只能是預先定義值的簡短清單之一，因此它們是列舉，而不是真正的字串。 某些列舉字串也會成對使用。

#### <a name="enumeration-pair"></a>列舉組

提供為列舉對的維度會使用下列格式顯示：

 _\<enumeration value from one end point\> : \<enumeration value from the other endpoint\>_

列舉值的順序是一致的，但無法反映第一或第二個端點的順序。

例如，網路連接詳細資料組會顯示兩個端點的網路連接詳細資料值：

|網路連接詳細資料配對 |如何解譯 |
|:--- |:--- |
|有線 ：有線 |第一個端點和第二個端點都使用有線乙太網路連接。 |
|有線 ：wifi |第一個端點使用有線乙太網路Wi-Fi，第二個端點則使用有線乙太網路連接，第一個端點則Wi-Fi連接。 |
|：wifi |第一個端點使用 WiFi 連接，而第二個端點所使用的網路連接不明，或第二個端點使用 WiFi 連接，而第一個端點所使用的網路連接是未知的。 |
| | |

#### <a name="blank-values"></a>空白值

下表列出維度可能空白的可能原因。 如果 QoE 記錄可用維度為 False，則許多維度和度量單位都會空白。 當通話未成功建立，或用戶端無法將其遙測傳送至服務時，通常會發生這種情況。

### <a name="notes-on-measurements"></a>度量的注意事項

#### <a name="accuracy-limitations"></a>精確度限制
特定使用者和通話計數測量仰賴對資料集執行不同的計數運算，以計算計數。 根據執行作業的列數，不同 countif 作業目前內生的錯誤多達 0.2%。 針對最精確的音量，您應該使用串流計數量詞，因為它們不仰賴這個不同的 countif 運算。 篩選以降低資料量可能會減少錯誤，但可能無法排除不同通話和使用者計數中的錯誤來源。 有關此限制的資訊，請參閱 [dcount 匯總函數](/azure/data-explorer/kusto/query/dcount-aggfunction)。

## <a name="filters"></a>過濾 器

許多維度和度量值也可以做為篩選。 您可以使用查詢中的篩選來排除資訊，方法與選取維度或度量單位以在查詢中新增或包含資訊的方式相同。
