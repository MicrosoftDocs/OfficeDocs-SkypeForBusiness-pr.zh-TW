<span data-ttu-id="0e213-211">在 CQD 中, 當 ICE 連接失敗或未報告計算資料流程分類所需的所有衡量標準時, 資料流程會標示為未分類。</span><span class="sxs-lookup"><span data-stu-id="0e213-211">In CQD, a stream is marked unclassified when ICE connectivity fails or when all the metrics required to compute the stream classification are not reported.</span></span>

<span data-ttu-id="0e213-212">若要檢查 ICE 連線失敗, 請檢查「第一個連通性 Ice」和「第二個連通性 Ice」的維度, 看看「失敗」的值。</span><span class="sxs-lookup"><span data-stu-id="0e213-212">To check for ICE connectivity failures, examine the dimensions "First Connectivity Ice" and "Second Connectivity Ice" for a "FAILED" value.</span></span> <span data-ttu-id="0e213-213">如果其中一個值表示失敗, 則會將此資料流程標示為未分類。</span><span class="sxs-lookup"><span data-stu-id="0e213-213">If either value indicates a failure, the stream will be marked as unclassified.</span></span>

<span data-ttu-id="0e213-214">如果未分類資料流程的 ICE 連線成功, 則該資料流程可能會被視為未分類, 因為未報告主要資料流程度量單位。</span><span class="sxs-lookup"><span data-stu-id="0e213-214">If ICE connectivity succeeded for an unclassified stream, the stream is likely considered unclassified because key stream metrics were not reported.</span></span> <span data-ttu-id="0e213-215">下列幾個原因可能不會報告這些指標:</span><span class="sxs-lookup"><span data-stu-id="0e213-215">There are a few reasons these metrics may not be reported:</span></span>

- <span data-ttu-id="0e213-216">**未收到 QoE 報表**-用於分類的度量單位會在通話結束時傳送的 QoE 報告中報告。</span><span class="sxs-lookup"><span data-stu-id="0e213-216">**QoE reports were not received** - The metrics used for classification are reported in a QoE report sent at the end of a call.</span></span> <span data-ttu-id="0e213-217">如果未產生此報告 (例如, 由於某些協力廠商端點可能無法傳送 QoE) 或無法傳送 (例如網路中斷), CQD 無法將資料流程分類。</span><span class="sxs-lookup"><span data-stu-id="0e213-217">If this report is not produced (e.g., because some third-party endpoints may not send QoE) or was not able to be sent (e.g., because of a network outage), CQD is unable to classify the stream.</span></span>

> [!TIP]
> <span data-ttu-id="0e213-218">您可以使用「QoE 記錄可用」維度來判斷是否已收到資料流程的 QoE 報告。</span><span class="sxs-lookup"><span data-stu-id="0e213-218">The "QoE Record Available" dimension can be used to determine whether a QoE report was received for a stream.</span></span> <span data-ttu-id="0e213-219">請注意, 如果 QoE 報告是從任一端點接收, 此維度的值會是 "True"。</span><span class="sxs-lookup"><span data-stu-id="0e213-219">Note that this dimension will have a value of "True" if a QoE report was received from either endpoint.</span></span> <span data-ttu-id="0e213-220">您必須使用兩個端點的 QoE 報告, 才能精確報告指標。</span><span class="sxs-lookup"><span data-stu-id="0e213-220">A QoE report from both endpoints is required for the most accurate reporting of metrics.</span></span>

- <span data-ttu-id="0e213-221">**短通話**-短時間的通話可能沒有足夠的媒體活動來計算金鑰資料流程度量單位。</span><span class="sxs-lookup"><span data-stu-id="0e213-221">**Short calls** - Short calls may not have enough media activity to compute key stream metrics.</span></span> <span data-ttu-id="0e213-222">若沒有這些指標, CQD 就無法將資料流程分類。</span><span class="sxs-lookup"><span data-stu-id="0e213-222">Without these metrics, CQD is unable to classify the stream.</span></span>

> [!TIP]
> <span data-ttu-id="0e213-223">[持續時間 (秒)]、"Duration (分鐘)"、"持續時間5秒或更短" 以及 [持續時間60秒或更多] 可用於決定資料流程的持續時間。</span><span class="sxs-lookup"><span data-stu-id="0e213-223">The dimensions "Duration (Seconds)", "Duration (Minutes)", "Duration 5 seconds or less", and "Duration 60 seconds or more" can be used to determine the duration of a stream.</span></span> <span data-ttu-id="0e213-224">[平均通話持續時間] 度量單位也可以用來計算一組資料流程的平均持續時間。</span><span class="sxs-lookup"><span data-stu-id="0e213-224">The measurement "Avg Call Duration" can also be used to compute the average duration for a set of streams.</span></span>

- <span data-ttu-id="0e213-225">**低資料包利用率**-例如「短通話」案例, 需要足夠的資料包使用量才能計算重要資料流程度量單位。</span><span class="sxs-lookup"><span data-stu-id="0e213-225">**Low packet utilization** - Like the "short call" scenario, sufficient packet utilization is required for computation of key stream metrics.</span></span> <span data-ttu-id="0e213-226">若沒有這些指標, CQD 就無法將資料流程分類。</span><span class="sxs-lookup"><span data-stu-id="0e213-226">Without these metrics, CQD is unable to classify the stream.</span></span>
    - <span data-ttu-id="0e213-227">常見的低資料包利用率情況是, 當使用者加入會議來收聽簡報者, 但卻沒有說話時 (可能會讓麥克風在大部分通話中靜音)。</span><span class="sxs-lookup"><span data-stu-id="0e213-227">A common low packet utilization scenario occurs when a user joins a meeting to listen to the presenter but never speaks (likely muting the microphone for most of the call).</span></span> <span data-ttu-id="0e213-228">在這種情況下, 一個音訊資料流程會有大量的資料包利用率 (入站至用戶端), 而另一資料流程幾乎不需要任何資料包利用率 (從用戶端輸出)。</span><span class="sxs-lookup"><span data-stu-id="0e213-228">In such a scenario, one audio stream will have high packet utilization (inbound to the client) while the other will have little to no packet utilization (outbound from the client).</span></span> <span data-ttu-id="0e213-229">在這種情況下, 資料流程的持續時間可能是一個小時或更長, 但從用戶端到伺服器的資料流程資料包利用率會因麥克風靜音而極低, 導致未分類的資料流程。</span><span class="sxs-lookup"><span data-stu-id="0e213-229">In this scenario, the duration of the stream may be an hour or longer but the packet utilization on the stream from the client to the server will be extremely low due to the microphone being muted, resulting in an unclassified stream.</span></span>

> [!TIP]
> <span data-ttu-id="0e213-230">[資料包利用率] 維度和 "Avg Packet 利用率" 測量值可用來判斷資料流程的資料包活動。</span><span class="sxs-lookup"><span data-stu-id="0e213-230">The "Packet Utilization" dimension and "Avg Packet Utilization" measurement can be used to determine the packet activity of a stream.</span></span>
