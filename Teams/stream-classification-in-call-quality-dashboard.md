## <a name="unclassified-streams"></a><span data-ttu-id="06f18-211">未分類資料流程</span><span class="sxs-lookup"><span data-stu-id="06f18-211">Unclassified Streams</span></span>

<span data-ttu-id="06f18-212">在 CQD 中，如果互動式連線建立（ICE）連線失敗，或未報告計算資料流程分類所需的所有衡量標準，資料流程會標示為未_分類_。</span><span class="sxs-lookup"><span data-stu-id="06f18-212">In CQD, a stream is marked _Unclassified_ when Interactive Connectivity Establishment (ICE) connectivity fails or when all the metrics required to compute the stream classification are not reported.</span></span>

<span data-ttu-id="06f18-213">若要檢查 ICE 連線失敗，請檢查「第一個連通性 Ice」和「第二個連通性 Ice」的維度，看看「失敗」的值。</span><span class="sxs-lookup"><span data-stu-id="06f18-213">To check for ICE connectivity failures, examine the dimensions "First Connectivity Ice" and "Second Connectivity Ice" for a "FAILED" value.</span></span> <span data-ttu-id="06f18-214">如果其中一個值代表失敗，該資料流程會標示為未_分類_。</span><span class="sxs-lookup"><span data-stu-id="06f18-214">If either value indicates a failure, the stream is marked as _Unclassified_.</span></span>

<span data-ttu-id="06f18-215">如果未_分類_資料流程的 ICE 連線成功，則該資料流程可能會被視為未_分類_，因為未報告主要資料流程度量單位。</span><span class="sxs-lookup"><span data-stu-id="06f18-215">If ICE connectivity succeeded for an _Unclassified_ stream, the stream is likely considered _Unclassified_ because key stream metrics were not reported.</span></span> <span data-ttu-id="06f18-216">下列幾個原因可能不會報告這些指標：</span><span class="sxs-lookup"><span data-stu-id="06f18-216">There are a few reasons these metrics may not be reported:</span></span>

- <span data-ttu-id="06f18-217">**未收到 QoE 報告**：用於分類的度量單位會在通話結束時傳送的 QoE 報告中報告。</span><span class="sxs-lookup"><span data-stu-id="06f18-217">**QoE reports were not received** — The metrics used for classification are reported in a QoE report sent at the end of a call.</span></span> <span data-ttu-id="06f18-218">如果未產生此報告（例如，由於某些協力廠商端點可能無法傳送 QoE）或無法傳送（例如，由於網路中斷），CQD 無法將資料流程分類。</span><span class="sxs-lookup"><span data-stu-id="06f18-218">If this report is not produced (for example, because some third-party endpoints may not send QoE) or could not be sent (for example, because of a network outage), CQD is unable to classify the stream.</span></span>

> [!TIP]
> <span data-ttu-id="06f18-219">您可以使用「QoE 記錄可用」維度來判斷是否已收到資料流程的 QoE 報告。</span><span class="sxs-lookup"><span data-stu-id="06f18-219">The "QoE Record Available" dimension can be used to determine whether a QoE report was received for a stream.</span></span> <span data-ttu-id="06f18-220">請注意，如果 QoE 報告是從任一端點接收，此維度的值會是 "True"。</span><span class="sxs-lookup"><span data-stu-id="06f18-220">Note that this dimension will have a value of "True" if a QoE report was received from either endpoint.</span></span> <span data-ttu-id="06f18-221">您必須使用兩個端點的 QoE 報告，才能精確報告指標。</span><span class="sxs-lookup"><span data-stu-id="06f18-221">A QoE report from both endpoints is required for the most accurate reporting of metrics.</span></span>

- <span data-ttu-id="06f18-222">**短通話**：短時間的通話可能沒有足夠的媒體活動來計算金鑰資料流程度量單位。</span><span class="sxs-lookup"><span data-stu-id="06f18-222">**Short calls** — Short calls may not have enough media activity to compute key stream metrics.</span></span> <span data-ttu-id="06f18-223">若沒有這些指標，CQD 就無法將資料流程分類。</span><span class="sxs-lookup"><span data-stu-id="06f18-223">Without these metrics, CQD is unable to classify the stream.</span></span>

> [!TIP]
> <span data-ttu-id="06f18-224">[持續時間（秒）]、"Duration （分鐘）"、"持續時間5秒或更短" 以及 [持續時間60秒或更多] 可用於決定資料流程的持續時間。</span><span class="sxs-lookup"><span data-stu-id="06f18-224">The dimensions "Duration (Seconds)", "Duration (Minutes)", "Duration 5 seconds or less", and "Duration 60 seconds or more" can be used to determine the duration of a stream.</span></span> <span data-ttu-id="06f18-225">[平均通話持續時間] 度量單位也可以用來計算一組資料流程的平均持續時間。</span><span class="sxs-lookup"><span data-stu-id="06f18-225">The measurement "Avg Call Duration" can also be used to compute the average duration for a set of streams.</span></span>

- <span data-ttu-id="06f18-226">**低資料包利用率**（例如「短通話」案例）需要足夠的資料包利用率，才能計算重要資料流程度量單位。</span><span class="sxs-lookup"><span data-stu-id="06f18-226">**Low packet utilization** — Like the "short call" scenario, sufficient packet utilization is required for computation of key stream metrics.</span></span> <span data-ttu-id="06f18-227">若沒有這些指標，CQD 就無法將資料流程分類。</span><span class="sxs-lookup"><span data-stu-id="06f18-227">Without these metrics, CQD is unable to classify the stream.</span></span>
  - <span data-ttu-id="06f18-228">常見的低資料包利用率案例會在出席者加入會議來收聽簡報者，但不會在說話時（麥克風在大部分通話中為靜音）。</span><span class="sxs-lookup"><span data-stu-id="06f18-228">A common low packet utilization scenario occurs when an attendee joins a meeting to listen to the presenter, but never speaks (the microphone is muted for most of the call).</span></span> <span data-ttu-id="06f18-229">在這裡，入站至用戶端的音訊資料流程具有高資料包利用率，而從用戶端輸出的音訊資料流程幾乎沒有資料包利用率。</span><span class="sxs-lookup"><span data-stu-id="06f18-229">Here, the audio stream inbound to the client has high packet utilization while the audio stream outbound from the client has little to no packet utilization.</span></span> <span data-ttu-id="06f18-230">資料流程的持續時間可能是一個小時或更長，但從用戶端到伺服器的資料流程中的資料包利用率較低，因為麥克風已靜音，且未_分類_的資料流程結果。</span><span class="sxs-lookup"><span data-stu-id="06f18-230">The duration of the stream may be an hour or longer but the packet utilization on the stream from the client to the server is low since the microphone was muted, and an _Unclassified_ stream results.</span></span>
