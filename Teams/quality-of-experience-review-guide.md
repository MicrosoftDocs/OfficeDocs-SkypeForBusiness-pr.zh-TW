# <a name="quality-of-experience-review-guide"></a>體驗品質的審查指南

<!-- Note that this link to the Word doc is intentionally NOT the aka.ms/qerquide link -->
本指南針對 Microsoft 團隊和商務用 Skype Online 的 [磁片磁碟機值] 階段。 您可以下載本指南的[Word 版本](https://github.com/MicrosoftDocs/OfficeDocs-SkypeForBusiness/blob/live/Teams/downloads/quality-of-experience-review-guide.docx?raw=true)。

## <a name="introduction"></a>簡介

若要對改善使用者體驗帶來最大的影響，組織必須 operationalize 下圖所示的主要區域。 其他區域包括識別操作工作、針對品質指標建立目標、ascertaining 度量組織成功的標準，以及視需要將調查區域縮小。


![使用者體驗品質的主要區域](media/qerguide-image-keyareas.png "使用者經驗品質的主要區域包括音訊、可靠性、使用者問卷、裝置和用戶端。")

_圖 1-本指南涵蓋的主要操作區域_

透過連續評估與修正本指南中所述的區域，您可以減少對使用者體驗品質產生負面影響的可能性。 在部署中遇到的大多數使用者體驗問題，都可以分為下列類別：

-   防火牆或 proxy 配置不完整
-   低 wi-fi 覆蓋範圍
-   頻寬不足
-   點對點
-   用戶端版本與驅動程式不一致或過時
-   未優化或內建音訊裝置
-   有問題的子網或網路裝置

在部署團隊或商務用 Skype Online 前，請先進行適當的規劃與設計，您就可以減少維持高品質體驗所需的工作量。

本指南的重點是使用 [通話品質儀表板] （CQD）線上作為主要工具來報告及調查每一個區域，並在音訊上特別強調最大化採納與影響。 針對改善音訊體驗的網路所做的任何改善，也會直接翻譯成影片與桌面共用的改良功能。

為加速您的評估，提供[兩個策劃 CQD 範本](https://aka.ms/qertemplates)：一個用於管理所有網路，而另一個則只針對受管理（內部）網路篩選。 雖然所有網路範本報告都設定為顯示建立和網路資訊，但是當您在收集與上傳資訊時，仍然可以使用它們。 將組建資訊上傳到 CQD，可讓服務在與外部子網區分內部的同時，透過新增自訂的建築物、網路和位置資訊來加強報告。 如需詳細資訊，請參閱本指南稍後的[建立對應](#building-mapping)。

### <a name="intended-audience"></a>目標物件

本指南專供合作夥伴與客戶干係人使用，其中包含共同作業主管/架構師、顧問、變更管理/採納專家、支援/技術支援人員、網路組長、桌面組長及 IT 系統管理員等角色。

本指南也專供指定的品質擁護者使用。 如需詳細資訊，請參閱[品質擁護者角色](4-envision-plan-my-service-management.md#the-quality-champion-role)。

### <a name="service-metrics"></a>服務度量單位

服務度量單位是由特定的用戶端測量標準所組成。 在每次通話期間，用戶端會收集通話的遙測資訊，並在每次通話結束後提交報告，以便日後可透過 CQD 或[呼叫分析](https://techcommunity.microsoft.com/t5/Skype-for-Business-Blog/Introducing-Call-Analytics/ba-p/57309)進行存取。 這些指標包括：

-   較差的串流速率
-   設定失敗率
-   丟單失敗率


#### <a name="poor-stream-rate"></a>較差的串流速率

較差的串流速率（PSR.EXE）代表組織中品質較差的資料流程總百分比。 這個指標是要醒目提示貴組織在哪些區域中，您的組織能專注于減少此值並改善使用者體驗等最大的影響，這就是為什麼在查看 PSR.EXE 時， [managed 網路](#managed-vs-unmanaged-networks)是主要的焦點。 外部使用者也很重要，但調查會因組織而異。 考慮為外部使用者提供最佳做法，並獨立于整體組織調查外部通話。

CQD 中的實際度量單位依工作負載而有所不同，但為了讓品質體驗進行評審，我們主要將重點放在_音訊不佳的百分比_測量中。 PSR.EXE 是由下表所述的五個網路度量值所組成。 針對要分類為較差的資料流程，只有一個公制需要超過定義的閾值。 如需有關資料流程分類程式的詳細資訊，請參閱[這篇文章](stream-classification-in-call-quality-dashboard.md)。

> [!Note]
> CQD 提供「不佳的原因 ...」 測量，以更清楚地瞭解哪個條件導致資料流程分類為較差。

#### <a name="setup-failure-rate"></a>設定失敗率

設定失敗率（亦即在 CQD 中的_呼叫設定失敗百分比_測量）是在通話開始時端點之間無法建立媒體路徑的串流數。

這代表無法建立的任何媒體資料流程。 根據此處測量的使用者經驗影響的嚴重性，目標是將此值縮小至盡可能接近零。 這個指標的高值在新部署中的防火牆規則不完整的情況下比較常見，但請務必定期觀看。

這個指標是透過取得失敗的資料流程總數除以提交成功呼叫詳細資料記錄（CDR）的總串流數來計算：

-   **設定失敗率**= 總的通話設定失敗資料流程計數/總 CDR 可用資料流程計數

#### <a name="drop-failure-rate"></a>丟單失敗率

在 CQD 中，擊落失敗率（亦稱為 [_總呼叫中斷失敗百分比_]）是已成功建立的資料流程在媒體路徑無法正常終止的百分比。

這代表任何意外終止的媒體資料流程。 雖然這種影響並不像是無法設定的資料流程那樣嚴重，但會對使用者體驗造成負面影響。 突然且頻繁的媒體跌落不只會對使用者體驗產生嚴重影響，因此它們會導致使用者需要重新連線，進而導致生產率損失。

指標的計算方式為：將丟棄的資料流程總數除以已成功設定的資料流程總計數：

-   **丟單失敗率**= 總的呼叫中斷串流總數/總呼叫設定成功的資料流程計數

### <a name="user-experience"></a>使用者體驗

分析使用者體驗的美工圖案比科學更多，因為這裡收集的度量單位並不一定代表網路或服務發生問題，而是只指示使用者感覺問題。 Microsoft 提供內建的調查機制（稱為「評定我的通話比率（RMC）」），協助估量總體使用者體驗。 RMC 將會協助您從使用者的觀點回答下列問題：

-   我知道如何使用此方案嗎？
-   這個方案便於使用且直觀，而且它支援我的日常通訊需求？
-   此方案可協助我完成工作嗎？
-   這個方案的整體認識是什麼？
-   無論是在何處，都能在任何時間點使用解決方案？
-   我可以設定及維護通話嗎？

#### <a name="rate-my-call"></a>評價我的通話 

[評級我的通話] （RMC）會內置在團隊和商務用 Skype 中，而且會自動設定為在每10個通話中有一人或10% 時顯示給參與者。 此簡短問卷會要求使用者對通話進行評分，並提供一些內容，以瞭解為何通話品質可能較差。 一或兩個評等不佳，3到4個是良好的，5是極好的。 雖然這是延遲指標的一部分，但對於發現服務規格可能遺漏的問題而言，這是一個非常實用的規格。

> [!Note]
> 除非使用者透過提供良好的意見反應來回應 RMC 問卷，否則回應通常會傳回 overwhelmingly 否定。 大多數的使用者只會在通話品質較差時回應。 因此，即使服務度量良好，您的 RMC 報告也可能會在差側扭曲。

您可以使用 CQD 來報告 RMC 的使用者回應，並將範例報告包含在 CQD 範本中。 不過，本指南中不會詳細討論它們。 如需在商務用 Skype Online 中 RMC 的詳細資訊，以及教育使用者提供實用 RMC 回應的指導方針，請參閱[這篇博客文章](https://blogs.technet.microsoft.com/jenstr/2015/05/05/rate-my-call-in-skype-for-business-2015/)。

#### <a name="client-and-device-readiness"></a>用戶端和裝置準備就緒

您需要實體用戶端和裝置策略，以協助確保您的使用者擁有一致且積極的使用者體驗。 幾個主要原則會推動每個準備策略。

##### <a name="client-readiness"></a>用戶端就緒

強有力的用戶端準備策略可確保您的使用者能夠在享有最佳體驗的情況下執行最新版的用戶端。 Microsoft 定期為商務用 Skype 用戶端修補程式;確保您的環境中的最新資料保持在最新狀態，對於您的整體成功至關重要。 請務必記住修補網路、影片、USB 和音訊驅動程式，因為它們經常被忽視，且可能會影響使用者的體驗。 考慮將網路、Wi-fi、影片、USB 和音訊驅動程式新增到您目前的修補程式管理程式。

我們建議您不要讓您的用戶端版本低於六個月。 如果您使用的是 Office 隨選即用，就表示您已經隨時掌握服務的最新狀態。 使用隨附的[用戶端版本](#client-versions)（如本指南稍後所述）來協助您處理此程式。 您也可以利用 [通話比率] 範例報告，進一步加強用戶端就緒戰略。

> [!IMPORTANT]
> 目前，團隊用戶端會透過 Azure 內容傳遞網路自動發佈並更新，並將由服務保持最新狀態。 因此，用戶端就緒性與調查活動不適用於小組。


##### <a name="device-readiness"></a>裝置就緒

任何單一策略都不會影響使用者體驗，而不是您的裝置準備方案。 大多陣列織都很樂意從使用者移除不必要的裝置（例如，手機或其他專用音訊裝置），而這通常是切換至團隊或商務用 Skype 的核心業務調整。 不過，這些相同的組織有時候也能提供取代裝置，即使這些裝置成本較低也一樣。 使用內建麥克風和喇叭的新式膝上型電腦和電腦，不會針對商務用語音傳遞 IP （VoIP）進行優化。 這通常會為所有參與者產生不佳的體驗，尤其是當喇叭在嘈雜環境中時。 Microsoft 的裝置認證計畫可確保使用者使用任何適用于團隊或商務用 Skype 認證的裝置參與電話撥打電話時，它會產生優於未認證裝置的體驗。 

我們總建議在透過桌面用戶端參與語音通話時，小組和商務用 Skype 使用者必須使用經認證的耳機或喇叭。 如需 Microsoft 認證裝置的詳細資訊，請參閱這些有關[認證計畫](/SkypeForBusiness/certification/overview)的文章，並查看[合作夥伴解決方案目錄](https://partnersolutions.skypeforbusiness.com/solutionscatalog/personal-peripherals-pcs)。 使用本指南稍後所述的 [[裝置] 報告](#devices)，協助您管理裝置。


### <a name="categories-of-quality"></a>品質類別

Operationalizing 高品質且可靠的部署的成功與否，取決於您的組建操作嚴格程度。 具體來說，請特別注意下列圖表所示的三種類別。以下是本指南的重點：

#### <a name="service-management-tasks"></a>服務管理工作

在雲端-第一個世界，您必須執行特定的服務管理工作，以維持高品質的使用者體驗。 這些工作的範圍包括：確保有足夠的頻寬可達到服務，而不需要飽和網際網路連結，請確認所有受管理的網路區域的服務品質（QoS）都已就緒，且最後是在防火牆上保持在[Office 365 IP 範圍](https://aka.ms/o365ips)的最上層。

另一個重要的區域就是監視您部署中流行的裝置，並推動認證裝置的使用，以提供最佳的使用者體驗。

### <a name="what-is-cqd"></a>什麼是 CQD？

您可以使用通話品質儀表板（CQD），深入瞭解使用團隊和商務用 Skype 服務所進行的通話品質。 CQD 的設計目的是協助商務用 Skype 與團隊管理員和網路工程師優化網路，並密切瞭解品質、可靠性和使用者體驗。 CQD 會在整個組織中查看匯總遙測，在這種情況下，整體模式可能會變得很明顯，讓員工能作出明智的評估並規劃修正活動，以最大限度地 CQD 提供可讓您深入瞭解整體品質、可靠性和使用者體驗的度量報告。

本指南將協助您瞭解 CQD 的核心概念，以協助您在改善使用者對團隊或商務用 Skype Online 的體驗時，取得最大的影響。 您可以在[附錄](#other-resources)中找到其他 CQD 資源。

### <a name="expectations-using-cqd"></a>使用 CQD 的預期

CQD 雖然對分析趨勢和子網很有用，但並不一定會針對特定案例提供特定原因。 請務必瞭解這一點，並在使用 CQD 時設定正確的預期：

-   CQD 不會針對每個案例提供根本原因。
-   CQD 不會包含電話系統或音訊會議串流。
-   CQD 將會根據趨勢來撥打進一步調查的區域。

### <a name="report-editions"></a>報表版本

在 CQD Online 中有兩個報表版本： [摘要] 和 [詳細]。 使用位於畫面頂端列中的下拉式功能表，開啟報表版本。 所選報告版本的名稱會顯示在畫面頂端。

-   摘要報告是靜態報表，且無法進行編輯、下載或匯出。 
-   詳細報告可完全自訂，而且可以下載至 CSV 檔案、匯出或克隆。

如需兩個版本之間差異的完整說明，請參閱[這篇文章](turning-on-and-using-call-quality-dashboard.md)。

-   **摘要報告**將重點分析品質趨勢與每日、每月和表格報告，以協助識別品質較差的子網。 這是您第一次登入 CQD Online 時的預設登陸頁面。

### <a name="managed-vs-unmanaged-networks"></a>受管理的與非託管的網路

根據預設，CQD 中的所有端點都會分類為外部。 一旦推出檔案，我們就可以開始查看受管理的端點資料。 如先前所述，CQD 中的網路定義為：

-   _受管理的網路_（通常稱為內部或內部）會受到組織的影響及控制。 這包括內部局域網、遠端 WAN 和 VPN。
-   _未受管理的網路_（通常稱為外部或外部）不會受到組織的影響或控制。 未受管理的網路範例是賓館或機場網路。

### <a name="dimensions-measures-and-filters"></a>維度、量值與篩選

標準格式的 CQD 查詢包含下列三個參數：

-   **維度：** 我要如何旋轉資料。

-   **測度：** 我想要報告的內容。

-   **篩選：** 我要如何減少查詢所傳回的資料集。

另一個要看的是，_維度_是分組函數，一個_量值_是我感興趣的資料，而 [_篩選_] 是我想要將結果縮小到與我的查詢相關的結果。

標準格式的查詢範例會**顯示「資料流程」 [Measure] （由子網 [Dimension]）來建立 6 [Filter]**。 如需詳細資訊，請參閱[CQD 中可用的維度與量值](https://aka.ms/cqd-dm)。

### <a name="first-vs-second"></a>第一個與第二個 

CQD 中的許多維度與量值會分類為第一或第二個。 CQD 不使用 [來電者/被叫方] 欄位 _，因為呼叫者與被_叫方之間有中間的步驟，所以它們已_重新命名。_ 下列邏輯決定所涉及的端點標示為「第一」：

-   如果是在資料流程或通話中參與伺服器，則**首先**會是伺服器端點（[會議服務器]、[中繼伺服器] 等）。

-   **第二個**將一直是用戶端端點，除非資料流程位於兩個伺服器端點之間。

-   如果兩個端點都是相同類型，則首先根據使用者代理程式類別的內部順序來選擇。 這可確保順序一致。

如需判斷第一個或第二個端點相同的詳細資訊，請參閱[CQD 中可用的維度與量值](https://aka.ms/cqd-dm)。

### <a name="stream-vs-call"></a>資料流程與撥號

您必須瞭解通話與資料流程間的差異，才能正確選擇您要在 CQD 中看到的維度或量值。 雖然 CQD 的主要焦點是位於資料流程上，但也提供以呼叫為基礎的度量單位。

-   **串流：**_資料流程_存在於兩個端點之間。 每個方向只有一個資料流程，且需要兩個數據流才能進行通訊。 資料流程對於調查建築物、網路或子網很有用。 在某些情況下，會在量化指標的名稱（例如，呼叫設定資料流程或呼叫丟棄的資料流程）中使用 call 和 stream。 這些專案仍會分類為串流。

-   **通話：**_通話_是來自所有參與者的所有資料流程的群組。 通話至少包含兩個數據流。 單一通話至少會有兩個端點，每個端點最少為一個串流。

如需有關維度或量值是參照至通話或資料流程的其他指導方針，請參閱[CQD 中可用的維度與量值](https://aka.ms/cqd-dm)

### <a name="good-poor-and-unclassified-calls"></a>良好、較差及未分類通話

通話會分類為良好、較差或未分類。 讓我們花點時間深入討論每一個專案。

-   **良好或不佳：**[良好或較差的通話] 包含一組完整的服務度量，由服務產生並接收完整的 QoE 報告。 [在本指南前面](#poor-stream-rate)所述，判斷資料流程是否正常或較差。

-   未**分類：** 未分類的資料流程不包含完整的一組服務度量單位。 這些可能是短通話（通常少於60秒），因為無法計算平均值，也不會產生 QoE 報告。 未分類呼叫最常見的原因，就是幾乎沒有資料包利用率。 例如，參與者在靜音和永不說話時加入會議。 參與者正在接收，但無法傳送媒體。 如果沒有傳送媒體，就不會有任何可供 CQD 用來分類端點的輸出媒體資料流程的指標。

如需有關資料流程分類程式的詳細資訊，請參閱[這篇文章](stream-classification-in-call-quality-dashboard.md)。

### <a name="common-subnets"></a>常見子網

常見子網是賓館、家用網路、熱點及類似區域所使用的專用子網。 這些子網難以進行會審，因為其廣泛使用。 如果您的組織使用其中一個常見的子網，我們建議您將該網路移至另一個子網。 這將使 CQD 的報告變得更容易。 請注意，[所有網路] 範本中的報告已設定為排除這些子網，以將其排除在品質較差的來源。 常見的子網定義如下：其影響會因組織而異。

-   10.0.0.0/24
-   192.168.0.0/24
-   192.168.1.0/24
-   192.168.2.0/24
-   172.20.10.0/24
-   192.168.43.0/24

調查使用共同子網的受管理網路時，您需要使用第二個反身本機 IP 維度來組成群組子網。 此維度包含端點的公用 IP 位址。

### <a name="filtering-reports"></a>篩選報表

提供的範本包括數個內建查詢和報表篩選。 下列各節說明在整個範本中使用的最常見篩選。

#### <a name="url-filter"></a>URL 篩選

您可以使用 URL 篩選來篩選特定維度的每個報表。 最常見的 URL 篩選是用來篩選報告，以排除同盟參與者遙測，或將焦點放在團隊或商務用 Skype Online 上。 我們建議您在使用篩選時，為它們加上書簽，以方便參考。 

如果您要修正受管理的建築物或網路（其中的同盟端點可能會影響您的報表），則從 CQD 報告排除同盟資料就很有用。

若要實現 URL 篩選，請在瀏覽器位址列中，將下列專案附加到 URL 的結尾：

```PowerShell
/filter/[AllStreams].[Second Tenant Id]\|[YOUR TENANT ID HERE]
```

範例  

```https://cqd.lync.com/cqd/#/1234567/2018-08/filter/[AllStreams].[Second Tenant Id]|[TENANTID]```

若要篩選團隊或商務用 Skype 的報表，請將下列專案附加到 URL 的結尾：

```PowerShell
/filter/[AllStreams].[Is Teams]|[TRUE | FALSE]
```

範例

```https://cqd.lync.com/cqd/#/1234567/2018-08/filter/[AllStreams].[Is Teams]|[TRUE]```


> [!NOTE]
> 上述 URL 範例只適用于視覺標記法。 請使用預設的 CQD 連結<https://cqd.lync.com>。

## <a name="building-mapping"></a>建立對應

在團隊或商務用 Skype Online 部署中，所有用戶端都是外部的。 也就是說，根據預設，所有用戶端都會在 CQD Online 中報告為外部，不論用戶端是否已連線到內部公司網路。

當您使用 CQD 時，您必須知道端點的位置，以及它是否已連線至您可以管理的網路，或是您無法管理的網路（假設您只能改善可管理的網路）。 透過將子網和組建資訊上傳到 CQD Online，您就能讓 CQD 判斷端點是否已連線至內部的公司/受管理網路或外部/非託管網路。

### <a name="building-data-file-structure"></a>建立資料檔案結構

您上傳的資料檔格式必須符合下列需求，才能在上傳前通過驗證檢查。

-   檔案必須是 TSV 檔案，這表示每個資料列都是以 Tab 字元分隔，或是以逗號分隔每個資料行的 CSV 檔案。

-   檔案不能大於 50 MB。

-   資料檔案的內容*不能包含表格標題*。 換句話說，資料檔案的第一行必須是實際資料，而不是欄標題（例如 "Network"）。

-   針對每個資料行，資料類型只能是 String、Number 或 Bool。 如果資料類型是 Number，則該值必須是一個數值;如果是 Bool，則該值必須是0或1。

-   針對每個資料行，如果資料類型是 String，則資料可以是空白（但必須以適當的分隔符號分隔，也就是 Tab 字元或逗號）。 這只會將該欄位指派為空白字串值。

-   每個資料列都必須有14個數據行。 每一欄都必須具有下表所述的資料類型，而且這些欄必須按照資料表中所列的順序。

| 欄名稱        | 資料類型 | 範例                   | 級    |

| NetworkName        | String    | 美國/西雅圖/西雅圖-海-1 | 必要\*  |

#### <a name="supernetting"></a>Supernetting

您可以使用 supernetting （通常稱為無類別的網域間路由（CIDR））來取代定義每個子網。 *Supernet*是多個子網的組合，可共用單一路由前置詞。 您可以使用 supernetted 位址，而不是為每個子網新增專案。 支援 Supernetting，但我們不建議使用它。

例如，Contoso 的行銷大樓是由下列子網組成：

-   10.1.0.0/24-第一層
-   10.1.1.0/24-第二層
-   10.1.2.0/24-第三層
-   10.1.3.0/24-第四層

您可以使用 supernetted 位址（在此範例中為 10.1.0.0/22），而不是為每個子網上新增專案。

-   Network = 10.1.0。0
-   網路範圍 = 22

在執行 supernetting 之前，請先考慮以下幾點：

-   Supernetting 只能在含有8位到28位元遮罩的子網對應中使用。

-   Supernetting 佔用較少的時間，但代價是減少資料的豐富程度。 假設有一個涉及子網200.1.2.0 的品質問題。 如果您已執行 supernetting，就不會知道子網的組建位置或網路類型（例如，lab）。 如果您已定義所有子網，以取得建築物和上傳的樓層位置資訊，您就可以看到這種差異。

-   務必確保 supernetted 位址正確無誤，而且不會捕捉不想要的子網。

-   在資料中找到192.168.0.0 是很常見的。 對於許多組織而言，這表示使用者是在家中。 對其他人而言，這是衛星辦公室的 IP 位址配置。 如果您的組織有使用這項設定的辦事處，請不要將它納入您的組建檔案中，因為很難使用通用子網來區分家用和內部網路。 請參閱本指南前面的有關[常見子網](#common-subnets)的章節。

> [!IMPORTANT]
> 網路範圍可以用來代表 supernet。 所有新的組建資料檔案上傳都將會檢查任何重迭的範圍。 如果您先前上傳的是組建檔案，您應該下載目前的檔案並再次上傳，以找出任何重迭並修正問題。 先前上傳的檔案中的任何交疊，可能會導致無法正確地將子網對應至報表中的建築物。

#### <a name="vpn"></a>點對點

用戶端傳送至 Office 365 的體驗品質（QoE）資料（也就是 CQD 資料來源的位置），包括 VPN 標誌。 CQD 將會看到此為第一個 VPN 和第二個 VPN 維度。 不過，此標記會依賴 VPN 廠商的報告來向 Windows 確認 VPN 網路介面卡已註冊為遠端存取配接器。 並非所有的 VPN 供應商都正確註冊遠端存取配接器。 因此，您可能無法使用內建的 VPN 查詢篩選。 在建立資訊檔案中，有兩種方法可容納 VPN 子網：

> [!IMPORTANT]

### <a name="updating-a-building-file"></a>更新組建檔案

在收集建立和子網資訊時，系統管理員會經常在一段時間內，在多個反覆運算中上傳檔案，並在新的子網變為可用時，新增其建築物資訊。 發生這種情況時，您必須重新上傳您的組建檔案。 此程式就像上一節所述的初始上傳一樣，還有一些例外狀況，如下一節所述。

> [!Important]
> 一次只能有一個建立檔。 多個建築物檔案不是累加的。

#### <a name="adding-net-new-subnets"></a>新增全新子網

在某些情況下，您需要將全新的子網新增至 CQD，而不是原本是您的網路拓朴的一部分。 若要新增 net 新建子網，請在 CQD 租使用者資料上傳入口網站中執行下列動作：

1.  編輯原始的組建檔案，並提供至少在取得網路新子網前一天的結束日期。
2.  如果您還沒有最新的複本，請下載原始檔案。
3.  將全新的子網附加至原始的組建檔案。
4.  按照上述相同程式上傳新修改的組建檔案，並在前一個建築物檔案結束後，將 [開始日期] 設定為一天。

#### <a name="updating-the-current-building-file"></a>更新目前的組建檔案

如果已上傳檔案，但是您需要新增遺失的子網，請在 CQD 租使用者資料上傳入口網站中執行下列動作：

1.  如果您還沒有最新的複本，請下載原始檔案。
2.  在 CQD 中移除目前的檔案。
3.  將新的子網附加至原始檔案。
4.  上傳組建檔案。 請務必先將開始日期設為至少八個月，以便 CQD 會處理歷史資料。

### <a name="missing-subnets"></a>遺失子網

1.  **安裝失敗：** 無法建立通話。

## <a name="quality-investigations"></a>品質調查

在組織中，評估音訊品質狀態的下一個步驟是調查不良的串流速率（PSR.EXE）、TCP 和 proxy 使用方式。 請務必記住，CQD 資料並不會提供特定的根本原因，而是提供可能的問題區域，以與適當的小組開始合作，以取得修正活動。 

> [!NOTE]
> 本指南並未涵蓋範本中包含的所有報表。不過，以下所述的調查方法仍適用于這些報表。 如需詳細資訊，請參閱個別報表的描述。 

### <a name="quality"></a>品質

PSR.EXE 百分比可用來指出組織是否為指定焦點區域的會議定義指標目標。 請務必注意，即使高層百分比位於定義的目標中，個別的子網或建築物可能不符合已定義的目標，因此需要進一步調查。 例如，如果整個音訊 PSR.EXE 百分比是4月的2% （符合樣本目標），個別的建築物和子網可能仍會有較差的體驗，這取決於這2% 的整體分佈情況。 

若要評估不良資料流程的百分比，請使用品質報告。 提供各種品質報告來審查整個、會議、二方、PSTN 通話、VPN 和會議室的規格。 提供每月、每週及每日報告以協助此處理程式。 每週和每日報告都受限於受管理的網路範本，以提高其效能並減少雜色。 

#### <a name="quality-trend-analysis"></a>品質趨勢分析

[趨勢報告] 會隨著時間顯示品質資訊，並用於協助識別並瞭解每個興趣領域內的品質趨勢。 如上所述，範本中有一些報告樹，可用於調查品質;會議、二方、PSTN 通話、VPN 和會議室。 針對分析品質而言，調查程式是一樣的。 不過，我們建議您先開始使用會議，因為會議品質的任何改善也會對所有其他區域產生積極的影響。 

> [!Note]

### <a name="tcp"></a>TCP-OUT

TCP 被視為 [回切傳輸]，而不是您想要用於即時媒體的主要傳輸。 因為 TCP 的全狀態性質，所以它是回切傳輸的原因。 例如，如果通話是在潛在的網路上進行，而媒體資料包則是在延遲後的數秒前（這是不再有用的資料包），這會使頻寬更糟，這可能會導致錯誤的情況。 這會讓音訊 healer 縫合及伸展音訊，從而產生可聽見的專案，通常是以抖動的形式出現。

此區段中的報表不會對良好與不良的資料流程產生任何差異。 考慮到 UDP 是可取的，報告會尋找使用 TCP 進行音訊、影片及影片的螢幕共用（VBSS）。 提供的串流速率較差是為了協助比較 UDP 品質與 TCP 品質，讓您能專注于最大限度的效果。 TCP 使用量主要是由不完整的防火牆規則所造成。 如需小組和商務用 Skype Online 防火牆規則的詳細資訊，請參閱[Office 365 url 與 IP 位址範圍](https://aka.ms/o365ips)。

> [!Important]
> 強烈建議您先上傳[有效的生成](#building-mapping)檔案，讓您可以在查看 TCP 使用方式時，快速區分外部資料流。

> [!Note]
> [音訊]、[影片] 和 [VBSS]，將所有偏好的 UDP 做為主要傳輸。 舊版 RDP 應用程式共用工作負載只使用 TCP。

我們[建議](proxy-servers-for-skype-for-business-online.md)您永遠都不要針對商務用 Skype 和小組（尤其是媒體流量）略過這些代理。 Proxy 不會讓商務用 Skype 更加安全，因為其流量已經過加密。 效能相關問題可透過延遲與資料包遺失來引進環境。 這類問題將會在即時資料流至關重要的情況下，使用音訊、影片和螢幕共用來產生負面體驗。

HTTP 使用的最常見原因是 proxy 中缺少例外規則。 使用所提供的建築物或子網，您可以快速判斷需要為媒體旁路設定哪些 proxy。

確認您的 proxy 中有必要的[Office 365 fqdn](https://aka.ms/o365ips)是列入白名單。

如需有關在商務用 Skype Online 和小組中使用 proxy 的詳細資訊，[請參閱本文](proxy-servers-for-skype-for-business-online.md)。

驅動高品質的使用者體驗的重要部分是，確保受管理的用戶端執行的是最新版本的商務用 Skype，除了可確保支援的音訊、影片、網路及 USB 驅動程式都是最新版本。 這提供數個優點，其中包括：

-   Microsoft 會持續進行整個產品的一般改善及優化。 確保使用者收到這些更新，會降低遇到已解決問題的風險。

### <a name="devices"></a>裝置

若要使用麥克風裝置報告，我們需要瞭解平均觀念分數（MOS）的概念。 MOS 是用來估量所察覺之音訊品質的黃金標準測量。 它是從0到5的整數評分來表示。

語音品質的所有測量單位都是人員如何感覺語音品質。 因為它受到人類的認知影響，所以本身就是主觀的。 主觀測試有數種不同的方法。 大多數的語音品質測量都是以絕對分類評級（ACR）刻度為基礎。

在 ACR 主觀測試中，有統計數量巨大的人員會以1（差）到5（良好）的比例來評定其體驗品質。 分數的平均值是 MOS。 產生的 MOS 取決於向群組所公開的經驗範圍，以及評級的經驗類型。

因為對即時通訊系統進行語音品質的主觀測試是不切實際的，所以 Microsoft 團隊和商務用 Skype 會使用高級演算法來產生 MOS 值，以客觀預測主觀測試的結果。

可用的一組 MOS 及相關的度量單位可提供透過音訊裝置傳送給使用者的體驗品質。 

您可以為擁有針對小組和商務用 Skype 認證的裝置提供使用者，以減少因裝置本身（例如，內建的膝上型電腦喇叭和麥克風）而遇到負面體驗的可能性。 如需詳細資訊，請參閱[認證計畫](/SkypeForBusiness/certification/overview)和[合作夥伴解決方案目錄](https://partnersolutions.skypeforbusiness.com/solutionscatalog/personal-peripherals-pcs)中的相關文章。

裝置報告是用來依音量與 MOS 分數（僅限音訊）來評估裝置使用量，而且可以在 [用戶端 & 裝置] 下的隨附範本中找到。 

> [!IMPORTANT]
> 除非您排除聯盟參與者資料，否則這些報告將包含來自聯盟端點的用戶端遙測。 若要排除同盟端點，您必須在您組織的[租使用者識別碼](#tenant-id)中，新增**第二個租使用者 id**的查詢篩選器。 或者，您也可以使用[URL 篩選](#url-filter)來排除同盟參與者遙測。

> [!NOTE] 
> 請務必將 [月份年份] 篩選調整為 [當月]。 選取 [**編輯**]，然後調整**月份年份**以儲存新的預設月份。

> [!Note]
> 您可能會注意到，當您查看此報告時，您會看到多次報告相同的裝置。 這是由於將裝置報告給 CQD 的方式所導致。 硬體與作業系統區域設定的差異會導致報告裝置資料的方式差異。

##### <a name="remediation"></a>修正

通常，您需要探索並階段找出未認證的裝置，並將它們取代為認證的裝置。 查看裝置報告時的一些考慮事項包括：

-   適用于團隊和商務用 Skype 的裝置是否已獲認證？ 
-   您可以透過 [[呼叫分析](https://techcommunity.microsoft.com/t5/Skype-for-Business-Blog/Introducing-Call-Analytics/ba-p/57309)] 識別特定裝置的使用者。 請檢查以確定他們擁有最新的裝置驅動程式，且其裝置未透過 USB 集線器或塢站連接。 
-   各種不同的驅動程式版本在使用中有多少不同？ 他們是定期進行修補嗎？ 確保定期修補音訊、影片和 Wi-fi 驅動程式，以協助將這些問題排除在品質問題的來源，並讓使用者體驗更具預見性和一致性。

##### <a name="audio"></a>音訊

下一個工作是判斷[認證音訊裝置](https://partnersolutions.skypeforbusiness.com/solutionscatalog/personal-peripherals-pcs)的整體使用量。 我們建議您至少80% 的所有音訊資料流程都使用經過認證的音訊裝置。 最佳做法是將麥克風裝置報告匯出至 Excel，以計算認證或核准裝置的使用方式。 組織通常會保留所有核准裝置的清單，因此篩選及排序資料應該相當簡單。

##### <a name="video"></a>顯示器

影片驅動程式也很重要，就是繼續更新。 確保經常修正的視訊卡，將無法排除視頻驅動程式，因為這是視頻串流品質較差的來源。 使用[經過驗證的影片裝置](https://partnersolutions.skypeforbusiness.com/solutionscatalog/personal-peripherals-pcs)將有助於確保使用者體驗順暢且高品質。 支援 H-p 原生編碼的視頻裝置是可取的，以減少在視訊會議期間的 CPU 使用量。

##### <a name="wi-fi"></a>Wi-fi

Wi-fi 驅動程式也必須在一般節奏上進行修補，而且應該包含在您的修補程式管理原則中。 您可以透過維護最新的 Wi-fi 驅動程式來修正許多品質問題。 如需優化 Wi-fi 基礎結構的詳細資訊，請參閱[這篇有關 wi-fi 規劃的文章](/skypeforbusiness/certification/networking-wifi)。
