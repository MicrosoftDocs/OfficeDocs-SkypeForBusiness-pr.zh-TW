#  <a name="use-cqd-to-manage-call-and-meeting-quality-in-microsoft-teams"></a>在 Microsoft 團隊中使用 CQD 管理通話與會議品質 

這篇文章將協助您使用 Microsoft 團隊通話品質儀表板 (CQD) ，協助您的團隊管理員或支援人員或技術支援人員。 我們的指導方針會強調音訊品質案例，因為您針對改善音訊體驗所做的任何網路改進都會轉化為影片和共用的改良功能。

本指南的主要內容是兩個[策劃 CQD 範本](https://aka.ms/QERtemplates)，我們建議您先下載它們，然後再逐步閱讀本文中的指導方針。

本文假設您已[設定 CQD](turning-on-and-using-call-quality-dashboard.md)。

## <a name="what-is-quality"></a>什麼是品質？

在這種情況下，品質是服務衡量標準與使用者經驗的結合。


### <a name="service-metrics"></a>服務度量單位

服務度量單位是由特定的用戶端測量標準所組成。 在每次通話期間，用戶端會收集通話的遙測，並在每次通話結束時提交報告，以便在 CQD 或每個[使用者的呼叫分析](set-up-call-analytics.md)中存取。 這些度量單位包括 (但不限於) ：

-   傳入及傳出) 的資料流程 (不佳
-   設定失敗率
-   丟單失敗率


#### <a name="poor-stream-rate"></a>較差的串流速率

較差的串流速率 (PSR.EXE) 代表組織中品質較差的資料流程總百分比。 這個指標是要醒目提示貴組織在哪些區域中，您的組織能專注于減少此值並改善使用者體驗等最大的影響，這就是為什麼在查看 PSR.EXE 時， [managed 網路](#managed-versus-unmanaged-networks)是主要的焦點。 外部使用者也很重要，但調查會因組織而異。 考慮為外部使用者提供最佳做法，並獨立于整體組織調查外部通話。

CQD 中的實際度量單位依工作量而異，但在本文中，我們主要將重點放在_音訊不佳的百分比_測量中。 PSR.EXE 是由下表所述的五個網路度量值所組成。 針對要分類為較差的資料流程，只有一個公制需要超過定義的閾值。 CQD 提供「不佳的原因 ...」測量，以更清楚地瞭解哪個條件導致資料流程分類為較差。 若要深入瞭解，請參閱[CQD 中的資料流程分類](stream-classification-in-call-quality-dashboard.md)。

> [!Note]
> CQD 提供「不佳的原因 ...」測量，以更清楚地瞭解哪個條件導致資料流程分類為較差。

##### <a name="why-do-we-prefer-to-use-streams-instead-of-calls"></a>為什麼我們想要使用串流而不是通話？

資料流程可讓我們知道來電的哪個特定腿不佳-外寄或撥入。 當您要查看通話分析是否有不佳的通話時，請判斷呼叫者的資料流程 (輸出) 或被叫方資料流程 (入站) 。 判斷哪些資料流程會影響通話品質，對於會議而言甚至更加重要。 如果您只是查看通話資料，您會看到某人參與了多少會議，但您不會看到哪些人是使用中的喇叭，但卻看不到您的螢幕共用。

通話資料可提供使用方式指標，但不一定會將您引導至無法正常通話品質的根本原因。 透過查看資料流程方向，您可以找出不在受管理網路上的通話數、非員工 (（例如，供應商或其他) 網路上的人員）的呼叫。 在這些情況下，如果其他人的網路連線較差，整個通話都會標示為不佳。 您無法對外部因素執行任何動作，所以這個資料並不是很有説明的。

資料流程方向也可協助您找出有問題的裝置或用戶端。

 - 例如，如果您的裝置預算有限，且想要為大量音訊使用者提供裝置，請使用 [音訊使用量報告] (VoIP) ，然後篩選出站資料流程與會議。 尋找在內建麥克風說話的大量音訊使用者-這些專案可能會與較差的通話品質 (，而且您可能會想要為這些人) 提供音訊裝置。 為了更清楚起見，您可以篩選資料包利用率，讓您針對特別高容量的音訊使用者進行設定。 

  - 另一個範例涉及螢幕共用。 如果客戶使用舊的團隊用戶端，螢幕共用效能可能會受到影響。 您可以針對完成許多螢幕共用的人員，為用戶端升級劃分優先順序，以解決這個問題。

 - 透過找出資料流程的哪個方向造成不佳的通話品質，您可以判斷您是否有與 QoS 或頻寬相關的問題。 如果您尚未完全實現 QoS，或者您只在用戶端（而不是在輸入資料流程）標示資料包，您可能會看到較差的通話品質。 透過查看資料流程方向，您可以更精確地查看資料包遺失、延遲或抖動的特定方向。 

   - 例如，假設您在有線連線 (抖動) 中，讓使用者 complains 機器人音訊。 透過查看資料流程和方向，您可以判斷問題只會出現在進貨資料流程上，只適用于一組特定的子網。 在您將此資訊提供給您的網路小組之後，他們就能向下追蹤，找出無法繞過媒體流量的錯誤配置 WAN 加速器。 網路小組重新配置 WAN 加速器之後，抖動就會消失，並改善通話品質。 


#### <a name="setup-failure-rate"></a>設定失敗率

設定失敗率（亦即在 CQD 中的_呼叫設定失敗百分比_測量）是在通話開始時端點之間無法建立媒體路徑的串流數。

這代表無法建立的任何媒體資料流程。 考慮到這個問題對使用者經驗造成影響的嚴重性，目標是將這個值縮小至盡可能接近零。 這個指標的高值在新部署中的防火牆規則不完整的情況下比較常見，但請務必定期觀看。

這個指標是透過以下方式來計算：根據在 CDR) 中提交成功呼叫詳細資料 (記錄的總串流數來進行設定失敗的總串流數：

-   **設定失敗率**= 總的通話設定失敗資料流程計數/總 CDR 可用資料流程計數

#### <a name="drop-failure-rate"></a>丟單失敗率

在 CQD 中，擊落失敗率（亦稱為 [_總呼叫中斷失敗百分比_]）是已成功建立的資料流程在媒體路徑無法正常終止的百分比。

這代表任何意外終止的媒體資料流程。 雖然這種影響並非與設定失敗的資料流程一樣嚴重，但仍會對使用者體驗造成負面影響。 突然且頻繁的媒體跌落不只會對使用者體驗產生嚴重影響，因此它們會導致使用者需要重新連線，導致生產率損失 (不會提及不滿) 。

指標的計算方式為：將丟棄的資料流程總數除以已成功設定的資料流程總計數：

-   **丟單失敗率**= 總的呼叫中斷串流總數/總呼叫設定成功的資料流程計數

### <a name="user-experience"></a>使用者體驗

分析使用者體驗的美工圖案比科學更多，因為這裡收集的度量單位並不一定代表網路或服務發生問題，而是只指示使用者感覺問題。 CQD 包含內建的調查機制，可讓我的通話 (RMC) ，以協助估量整體使用者經驗。 RMC 可讓您深入瞭解使用者對下列問題的觀點：

-   我知道如何使用此方案嗎？
-   這個方案便於使用且直觀，而且它支援我的日常通訊需求？
-   此方案可協助我完成工作嗎？
-   這個方案的整體認識是什麼？
-   無論是在何處，都能在任何時間點使用解決方案？
-   我可以設定及維護通話嗎？

#### <a name="rate-my-call"></a>評價我的通話 

在小組和商務用 Skype 中建立 RMC) ， (的通話費率。 它會在每10個通話中自動彈出一個，或10%。 此簡短問卷會要求使用者對通話進行評分，並提供一些內容，以瞭解為何通話品質可能較差。 一或兩個評等不佳，3到4個是良好的，5是極好的。 雖然這是延遲指標的一部分，但對於發現服務規格可能遺漏的問題而言，這是一個非常實用的規格。

> [!Note]
> 人體因數：使用者通常會在通話品質良好時忽略調查，當通話品質不正確時，就會填寫問卷。 因此，即使服務衡量標準良好，您的 RMC 報告也可能會在差側扭曲。

您可以使用 CQD 來報告 RMC 的使用者回應，並將範例報告包含在 CQD 範本中。 不過，本文不會詳細討論。 

#### <a name="client-and-device-readiness"></a>用戶端和裝置準備就緒

您需要實體用戶端和裝置策略，以協助確保您的使用者擁有一致且積極的使用者體驗。 幾個主要原則會推動每個準備策略。

##### <a name="client-readiness"></a>用戶端就緒

讓團隊用戶端保持在最新狀態，可確保您的使用者永遠能取得最佳的體驗。 Microsoft 會頻繁發行[小組用戶端的更新](teams-client-update.md) (更新會在背景中自行安裝，除非您關閉此功能，但我們不建議) 。 請務必記住修補網路、影片、USB 和音訊驅動程式，因為它們經常被忽視，且可能會影響通話和會議品質。 考慮將網路、Wi-fi、影片、USB 和音訊驅動程式新增到您目前的修補程式管理程式。


##### <a name="device-readiness"></a>裝置就緒

任何單一策略都不會影響使用者體驗，而不是您的裝置準備方案。 例如，依賴其膝上型電腦和麥克風的使用者會在通話和會議中遇到許多背景雜音。 團隊專用於幾乎任何裝置，但如果您有與裝置相關的問題，請參閱[手機供團隊](phones-for-teams.md)使用。


### <a name="categories-of-quality"></a>品質類別

Operationalize 一組品質管制做法-這可讓您獲得良好通話和會議品質的最佳機會。 優質管理方案會解決下列類別：

-   **網路：** 注重資料流程比率較差的音訊品質 (PSR.EXE) 公制、TCP 使用量、有線和無線子網上，以及識別 HTTP proxy 與 VPN 的使用

-   **端點：** 音訊裝置和最新用戶端

-   **服務管理：** 此類別包含兩個區段：

    -   首先，Microsoft 必須負責管理和維護小組及商務用 Skype Online 服務。

    -   第二個是貴組織管理的工作，以確保對服務的可靠存取，例如更新建立資訊，並在新的 Office 365 IP 位址中維護防火牆，以將基礎結構新增至服務。

![組織中品質類別的圖形](media/qerguide-image-categories.png "組織中的品質類別：服務管理、端點和網路。")

請查看下列建議的工作清單，以維持品質。 您應該定期執行這些工作，例如 [每週]。

#### <a name="service-management-tasks"></a>服務管理工作

這些工作的範圍包括確保有足夠的頻寬可達到服務，而不需要飽和網際網路連結、驗證服務品質 (QoS) 在所有受管理的網路區域中都已就緒，且在防火牆上保持在[Office 365 IP 範圍](https://aka.ms/o365ips)的最上層。

#### <a name="network-tasks"></a>網路工作

網路任務有兩種類別：可靠性和品質。 可靠性側重于測量使用者成功進行呼叫並保持連線的能力。 [品質] 主要是在通話期間和結束後，透過使用者用戶端傳送給小組和商務用 Skype Online 的匯總遙測。 

根據可靠性對使用者體驗的重要影響，我們建議您先評估並調查可靠性量度，然後再深入瞭解品質。 

#### <a name="endpoints-tasks"></a>端點任務

此類別中的主要工作會移除一般[小組用戶端更新](teams-client-update.md)的任何障礙。 依預設，小組會定期自動更新 (除非您關閉該設定，否則我們不建議) 。 

您也應該監視裝置，並在您發現與裝置相關的問題時提供更新。

### <a name="expectations-using-cqd"></a>使用 CQD 的預期

使用通話品質儀表板 (CQD) ，深入瞭解使用團隊和商務用 Skype 服務所進行的通話品質。 CQD 的設計目的是協助小組和商務用 Skype 系統管理員和網路工程師優化網路，並密切瞭解品質、可靠性和使用者體驗。 CQD 會在整個組織中查看匯總遙測，其中的整體模式可能會變得很明顯;這可讓您做出明智的評定及規劃修正。 CQD 提供可讓您深入瞭解整體品質、可靠性和使用者體驗的度量報告。

-   CQD 不會針對每個案例提供根源
-   CQD 不包含電話系統或音訊會議串流
-   CQD 將會根據趨勢來撥打進一步調查的區域

### <a name="managed-versus-unmanaged-networks"></a>受管理的與非管理的網路

根據預設，CQD 中的所有端點都會分類為外部。 一旦推出檔案，我們就可以開始查看受管理的端點資料。 如先前所述，CQD 中的網路定義為：

-   _受管理的網路_（通常稱為內部或內部）會受到組織的影響及控制。 這包括內部局域網、遠端 WAN 和 VPN。
-   _未受管理的網路_（通常稱為外部或外部）不會受到組織的影響或控制。 未受管理的網路範例是賓館或機場網路。

### <a name="dimensions-measures-and-filters"></a>維度、量值與篩選

標準格式的 CQD 查詢包含下列三個參數：

-   **維度：** 我要如何旋轉資料。

-   **測度：** 我想要報告的內容。

-   **篩選：** 我要如何減少查詢所傳回的資料集。

另一個要看的是，_維度_是分組函數，一個_量值_是我感興趣的資料，而 [_篩選_] 是我想要將結果縮小到與我的查詢相關的結果。

標準格式的查詢範例會**顯示「資料流程」 [Measure] （由子網 [Dimension]）來建立 6 [Filter]**。 如需詳細資訊，請參閱[CQD 中可用的維度與量值](https://aka.ms/cqd-dm)。

### <a name="first-vs-second"></a>第一個與第二個 

CQD 中的許多維度與量值會分類為第一或第二個。 CQD 不使用 [來電者/被叫方] 欄位 _，因為呼叫者與被_叫方之間有中間的步驟，所以它們已_重新命名。_ 下列邏輯決定所涉及的端點標示為「第一」：

-   **首先**， (會議服務器、中繼伺服器等的伺服器端點，以及在資料流程或呼叫中包含伺服器時) 。

-   **第二個**將一直是用戶端端點，除非資料流程位於兩個伺服器端點之間。

-   如果兩個端點都是相同類型，則首先根據使用者代理程式類別的內部順序來選擇。 這可確保順序一致。

如需判斷第一個或第二個端點相同的詳細資訊，請參閱[CQD 中可用的維度與量值](https://aka.ms/cqd-dm)。

### <a name="stream-vs-call"></a>資料流程與撥號

您必須瞭解通話與資料流程間的差異，才能正確選擇您要在 CQD 中看到的維度或量值。 雖然 CQD 的主要焦點是位於資料流程上，但也提供以呼叫為基礎的度量單位。

-   **串流：**_資料流程_存在於兩個端點之間。 每個方向只有一個資料流程，且需要兩個數據流才能進行通訊。 資料流程對於調查建築物、網路或子網很有用。 在某些情況下，通話和資料流程是在量化指標的名稱中使用 (例如，呼叫設定資料流程或呼叫中斷串流) 。 這些專案仍會分類為串流。

-   **通話：**_通話_是來自所有參與者的所有資料流程的群組。 通話至少包含兩個數據流。 單一通話至少會有兩個端點，每個端點最少為一個串流。

如需有關維度或量值是參照至通話或資料流程的其他指導方針，請參閱[CQD 中可用的維度與量值](https://aka.ms/cqd-dm)

### <a name="good-poor-and-unclassified-calls"></a>良好、較差及未分類通話

通話會分類為良好、較差或未分類。 讓我們花點時間深入討論每一個專案。

-   **良好或不佳：**[良好或較差的通話] 包含一組完整的服務度量，由服務產生並接收完整的 QoE 報告。 [在本文先前](#poor-stream-rate)所述，判斷資料流程是否良好或變差。

-   未**分類：** 未分類的資料流程不包含完整的一組服務度量單位。 這些可能是短通話（通常少於60秒），因為無法計算平均值，也不會產生 QoE 報告。 未分類呼叫最常見的原因，就是幾乎沒有資料包利用率。 例如，參與者在靜音和永不說話時加入會議。 參與者正在接收，但無法傳送媒體。 如果沒有傳送媒體，就不會有任何可供 CQD 用來分類端點的輸出媒體資料流程的指標。

若要深入瞭解，請參閱[CQD 中的資料流程分類](stream-classification-in-call-quality-dashboard.md)。

### <a name="common-subnets"></a>常見子網

常見子網是賓館、家用網路、熱點及類似區域所使用的專用子網。 這些子網難以進行會審，因為其廣泛使用。 如果您的組織使用其中一個常見的子網，我們建議您將該網路移至另一個子網。 這將使 CQD 的報告變得更容易。 請注意，[所有網路] 範本中的報告已設定為排除這些子網，以將其排除在品質較差的來源。 常見的子網定義如下：其影響會因組織而異。

-   10.0.0.0/24
-   192.168.0.0/24
-   192.168.1.0/24
-   192.168.2.0/24
-   172.20.10.0/24
-   192.168.43.0/24

調查使用共同子網的受管理網路時，您需要使用第二個反身本機 IP 維度來組成群組子網。 此維度包含端點的公用 IP 位址。


## <a name="reliability-investigations"></a>可靠性調查

改善品質的第一個步驟是評估整個組織的可靠性狀態。 因為可靠性對於積極的使用者體驗至關重要，所以我們從測量可靠性的兩個元件開始：

1.  **安裝失敗：** 無法建立通話。

2.  **Drop 失敗：** 通話已建立且意外終止。

在本節中，我們將說明同時調查兩個區域的方法。

> [!NOTE]
> 本文並未涵蓋範本中包含的所有報表。 不過，以下所述的調查方法仍適用。 如需詳細資訊，請參閱個別報表的描述。


### <a name="setup-failures"></a>設定失敗

首先，請優先處理此區域中的修正設定失敗問題，因為這些失敗會對使用者體驗產生嚴重的負面影響。

您可以評估組織的整體設定失敗百分比，然後根據組建或網路最高的百分比來排定調查區域的優先順序。 

#### <a name="setup-failure-trend-analysis"></a>設定失敗趨勢分析

此報告顯示串流的總總量、資料流程設定失敗，以及資料流程設定失敗率。 指向任何一個資料行，以顯示其個別的值。 

##### <a name="analysis"></a>分析

您可以使用此報告來回答下列問題，並決定您的下一個動作課程：

-   目前月份的電話設定失敗總百分比是什麼？

-   [通話設定失敗] 的總百分比是低於或高於定義的目標度量嗎？

-   失敗趨勢是否較差或比上個月更佳？

-   失敗趨勢是增加、穩定或遞減嗎？

不論這些問題的答案如何，請使用隨附子報表來進一步調查，以尋找任何可能需要修正的個別建築物或子網。 雖然整體失敗率可能低於目標度量，但一或多個建築物或網路的失敗率可能高於目標指標，且需要調查。

#### <a name="setup-failure-investigations"></a>設定失敗調查 

這個摘要報告是用來探索及隔離任何可能需要修正的建築物或網路。

> [!NOTE]
> 請務必將 [月年度] 報表篩選調整為 [本月]。 選取 [**編輯**]，然後調整 [**月**] 報表篩選，以儲存新的預設月份。

##### <a name="remediation"></a>修正 

將您的第一次修正工作重點放在具有最大大量故障的建築物或子網上。 這將會最大化對使用者體驗的影響，並有助於快速減少組織電話設定失敗的速度。 下表列出由 CQD 報告的設定失敗的兩個原因。

| 通話設定失敗的原因       | 常見的原因                    |
|----------------------------------|----------------------------------|
| 遺失固件深層資料包檢查例外規則 | 表示由於深入的資料包檢查規則而使媒體路徑無法建立，所以沿著路徑的網路裝置無法進行建立。 這可能是因為防火牆規則沒有正確設定。 在這種情況下，TCP 握手成功，但 SSL 握手沒有。      |
| 遺失的 FW IP 封鎖例外規則      | 表示路徑中的網路裝置無法建立媒體路徑給 Microsoft 365 或 Office 365 網路。 這可能是因為無法正確設定 proxy 或防火牆規則，以允許存取用於團隊和商務用 Skype 流量的 IP 位址和埠。 |

當您開始修正時，您可以將精力集中在特定的建築物或子網上。 如上表所示，這些問題是由防火牆或 proxy 設定所導致。 請參閱下表中的選項，以取得修正動作。

|      修正      |指導方針  |
|-----------------------|----------|
| 設定防火牆 | 與您的網路小組合作，並對照[Office 365 IP 位址清單](https://aka.ms/o365ips)驗證您的防火牆設定。<br><br>確認[媒體子網](https://support.office.com/article/Office-365-URLs-and-IP-address-ranges-8548a211-3fe7-47cb-abb1-355ea5aa88a2#bkmk_teams)和埠都包含在防火牆規則中。 <br><br>確認在防火牆中開啟[必要的埠](prepare-network.md)。 UDP 應該有優先權，因為 TCP 被視為音訊、影片和影片畫面共用的容錯回復通訊協定，而其使用會影響通話品質。 舊版 RDP 應用程式共用只使用 TCP。|

### <a name="drop-failures"></a>Drop 失敗

與 setup 失敗代碼不同的是，CQD 沒有 drop 失敗程式碼來指出為什麼會發生 drop 失敗，這會讓您難以隔離特定的根本原因。 若要更完善的會審放下失敗，請使用推理的方法。 透過修正媒體的任何感興趣區域、修補用戶端和驅動程式，以及推動小組和商務用 Skype 的認證裝置使用量，您可能會想要拒絕丟單失敗。

#### <a name="drop-failure-trend-analysis"></a>丟單失敗趨勢分析

此報告顯示音訊流量總量、總丟單失敗次數及擊落失敗率。 指向任何一個資料行，以顯示其值。 


##### <a name="analysis"></a>分析

透過使用這種類型的報表，您可以回答下列問題：

-   目前的擊落失敗率為何？
-   Drop 失敗率是否低於定義的目標度量？
-   失敗趨勢是否較差或比上個月更佳？
-   失敗趨勢是增加、穩定或遞減嗎？

無論上述問題的答案如何，請花時間調查子報表，以尋找任何可能需要修正的建築物或網路。 雖然整體 drop 失敗率可能低於目標度量，但一或多個建築物或網路的擊落失敗率可能高於目標指標，且需要調查。

#### <a name="drop-failure-investigations"></a>擊落失敗調查

在此報告的失敗指示此通話被意外丟棄，並產生消極的使用者體驗。 與趨勢報告不同，這些報告提供其他深入瞭解需要進一步調查之子網的深入見解。


##### <a name="remediation"></a>修正

使用隨附的表格報告，您可以隔離網路中超過您所定義之目標度量的問題區域。 將您的第一次修正工作重點放在具有最高總串流數的建築物或子網上，以產生最大的影響。

呼叫下降的常見原因：

-   預配的網路或網際網路出口
-   受限制的網路上沒有設定 QoS
-   舊版用戶端版本
-   使用者行為

在您探索問題區域之後，您可以使用[每個使用者的呼叫分析](use-call-analytics-to-troubleshoot-poor-call-quality.md)，進一步查看該組建中的特定問題的使用者。 [通話分析] 包含其他 EUII 資料，對於進一步產生 drop 失敗的可能原因也很有用。

無論下一個步驟為何，建議您的技術人員在特定建築物或子網中發現的問題都是一種很好的做法。 這可讓技術支援人員快速回應來電，並更有效率地會審使用者。 然後，您可以將已標記的使用者報告回到工程小組，進一步進行調查。

### <a name="quality"></a>品質

PSR.EXE 百分比可用來指出組織是否為指定焦點區域的會議定義指標目標。 請務必注意，即使高層百分比位於定義的目標中，個別的子網或建築物可能不符合已定義的目標，因此需要進一步調查。 例如，如果整個音訊 PSR.EXE 百分比是4月的2% （符合樣本目標），個別的建築物和子網可能仍會有較差的體驗，這取決於這2% 的整體分佈情況。 

若要評估不良資料流程的百分比，請使用品質報告。 提供各種品質報告來審查整個、會議、二方、PSTN 通話、VPN 和會議室的規格。 提供每月、每週及每日報告以協助此處理程式。 每週和每日報告都受限於受管理的網路範本，以提高其效能並減少雜色。

[趨勢報告] 會隨著時間顯示品質資訊，並用於協助識別並瞭解每個興趣領域內的品質趨勢。 如上所述，範本中有一些報告樹，可用於調查品質;會議、二方、PSTN 通話、VPN 和會議室。 針對分析品質而言，調查程式是一樣的。 不過，我們建議您先開始使用會議，因為會議品質的任何改善也會對所有其他區域產生積極的影響。 

> [!Note]
> 調查兩方、PSTN 通話和會議室與調查會議類似。 焦點是隔離具有最差品質的建築物或子網，並找出品質不佳的原因。

> [!Important]

將修正工作集中在具有最大大量資料流程的建築物或子網上，因為這可讓您獲得最大的影響，並有助於快速改善使用者體驗。 使用抖動、資料包遺失及往返時間 (RTT) 測量，以瞭解品質不佳 (可能會有一個以上的問題) ：

### <a name="tcp"></a>TCP-OUT 

傳輸控制通訊協定 (TCP) 被視為 [回切傳輸]，而不是您想要用於即時媒體的主要傳輸。 因為 TCP 的全狀態性質，所以它是回切傳輸的原因。 例如，如果通話是在潛在的網路上進行，而媒體資料包則是在延遲後的數秒前（這是不再有用的資料包），這會使頻寬更糟，這可能會導致錯誤的情況。 這會讓音訊 healer 縫合及伸展音訊，從而產生可聽見的專案，通常是以抖動的形式出現。

此區段中的報表不會對良好與不良的資料流程產生任何差異。 考慮到 UDP 是可取的，報告會尋找使用 TCP 進行音訊、影片及影片的螢幕共用 (VBSS) 。 提供的串流速率較差是為了協助比較 UDP 品質與 TCP 品質，讓您能專注于最大限度的效果。 TCP 使用量主要是由不完整的防火牆規則所造成。 如需小組和商務用 Skype Online 防火牆規則的詳細資訊，請參閱[Microsoft 365 和 Office 365 url 與 IP 位址範圍](https://aka.ms/o365ips)。

> [!Note]
> [音訊]、[影片] 和 [VBSS]，將所有偏好的 UDP 做為主要傳輸。 舊版 RDP 應用程式共用工作負載只使用 TCP。

雖然您想要盡可能地降低 TCP 使用量，但您可能會在其他健康的部署中看到一些 TCP 使用量。 TCP 本身不會對不佳的呼叫造成影響，因此會提供串流速度，以協助找出 TCP 使用量是否是品質不佳的參與者。 

#### <a name="tcp-investigations"></a>TCP 調查

在提供的 CQD 範本中，您可以使用 [Managed 網路] 或 [所有網路] 範本來建立和子網報告，流覽至 TCP 資料流程。 出於調查 TCP 使用的目的，程式是一樣的，因此我們將重點放在會議上的討論。


##### <a name="remediation"></a>修正

這個報告會識別對 TCP 使用量而言所造成的特定建築物與子網。 此外還包含額外的報告，以識別通話中所使用的 Microsoft 中繼 IP，以協助隔離遺失的防火牆規則。 將您的修正工作重點放在那些具有最高 TCP 資料流程的建築物上，以獲得最佳效果。

TCP 使用的最常見原因是防火牆或 proxy 中缺少例外規則。 我們將在下一節中討論 proxy，所以現在將焦點放在防火牆上。 使用所提供的建築物或子網，您可以判斷需要更新哪些防火牆。

| 修正        | 指導方針     |
|--------------------|--------------------------------------|
| 設定防火牆 | 確認已從您的防火牆中排除[Microsoft 365 或 Office 365 IP 埠和位址](https://aka.ms/o365ips)。 針對媒體相關的 TCP 問題，請將您的初始工作重點放在下列各項：<ul><li>確認用戶端媒體子網 13.107.64.0/18 和 52.112.0.0/14 都在您的防火牆規則中。</li><li>UDP 埠3478–3481是必要的媒體埠，必須開啟，否則用戶端會傳回 TCP 埠443的故障。</li></ul> |
| 是否             | 使用[Microsoft Network 評量工具](https://www.microsoft.com/download/details.aspx?id=53885)，檢查與特定 Microsoft 365 或 OFFICE 365 IP 位址的連線問題，以及受影響的建築物或子網上的埠。    |

### <a name="http-proxy"></a>HTTP proxy

出於許多原因，HTTP proxy 不是建立媒體會話的首選路徑。 許多包含可防止服務連線完成並帶來中斷的深資料包檢查功能。 此外，幾乎所有的 proxy 都會強制 TCP，而不是允許使用 UDP，這是針對最佳音訊品質所建議的。

我們總是建議您將用戶端設定為直接連線至團隊和商務用 Skype 服務。 這對於媒體通信量而言尤為重要。


> [!IMPORTANT]
> 我們建議您上傳[有效的組建](CQD-upload-tenant-building-data.md)檔案，讓您可以在分析 proxy 使用量時，區分外部音訊資料流程。 


#### <a name="http-proxy-usage"></a>HTTP proxy 使用量

範本這個區段中的 HTTP proxy 串流報告，與 TCP 報告非常類似。 它不會查看通話是否較差或良好，但通話是否是經由 HTTP 連線。

![使用 HTTP 之音訊資料流程報告的螢幕擷取畫面](media/qerguide-image-audiostreamswithhttp.png)

##### <a name="analysis"></a>分析

您想要查看盡可能少的 HTTP 媒體資料流程。 如果您有資料流程遍歷您的 proxy，請諮詢您的網路小組，以確保適當的排除措施正確無誤，讓用戶端直接路由至小組或商務用 Skype Online 媒體子網。

如果您的組織只有一個網際網路 proxy，請確認正確的[Microsoft 365 或 Office 365 url 與 IP 位址範圍排除](https://aka.ms/o365ips)。 如果您的組織中設定了多個網際網路 proxy，請使用 HTTP 子報表來隔離受影響的建築物或子網。

對於無法繞過 proxy 的組織而言，請務必將商務用 Skype 用戶端設定為在 proxy 之後正確登入，如商務用 Skype 中所述，請[使用 proxy 伺服器登入，而不是嘗試直接](https://support.microsoft.com/help/3207112/skype-for-business-should-use-proxy-server-to-sign-in-instead-of-tryin)連線。 


#### <a name="http-proxy-investigations"></a>HTTP proxy 調查

此報告會識別出對 HTTP 使用量有影響的特定建築物與子網。


##### <a name="remediation"></a>修正

我們[建議](proxy-servers-for-skype-for-business-online.md)您永遠都不要針對商務用 Skype 和小組（尤其是媒體流量）略過這些代理。 Proxy 不會讓商務用 Skype 更加安全，因為其流量已經過加密。 效能相關問題可透過延遲與資料包遺失來引進環境。 這類問題將會在即時資料流至關重要的情況下，使用音訊、影片和螢幕共用來產生負面體驗。

HTTP 使用的最常見原因是 proxy 中缺少例外規則。 使用所提供的建築物或子網，您可以快速判斷需要為媒體旁路設定哪些 proxy。

確認所需的[Microsoft 365 或 Office 365 fqdn](https://aka.ms/o365ips)是在您的 proxy 中列入白名單。

## <a name="endpoint-investigations"></a>端點調查

本節重點針對用戶端版本的報告工作，以及使用已認證的裝置。 您可以在用戶端版本、用戶端類型、捕獲裝置和驅動程式 (麥克風) 、視頻捕獲裝置以及 Wi-fi 供應商和驅動程式版本中使用報告。

> [!NOTE]
> 本文並未涵蓋範本中包含的所有報表。不過，以下所述的調查方法仍適用。 如需詳細資訊，請參閱個別報表的描述。

### <a name="client-versions"></a>用戶端版本

這些報告主要是識別使用中的商務用 Skype 用戶端版本，以及其在環境中的相對量。

> [!IMPORTANT]
> 目前，團隊用戶端會透過 Azure 內容傳遞網路自動發佈並更新，並將由服務保持最新狀態。 因此，您不需要 (監視團隊用戶端版本，除非您關閉自動更新，否則我們不建議) 。

除非您排除聯盟參與者資料，否則這些報告將包含來自聯盟端點的用戶端遙測。 若要排除同盟端點，您必須在您組織的[租使用者識別碼](CQD-data-and-reports.md#how-to-find-your-tenant-id)中，新增第二個租使用者 id 的查詢篩選器。 或者，您也可以使用[URL 篩選](CQD-data-and-reports.md#url-filters)來排除同盟參與者遙測。



#### <a name="remediation"></a>修正

驅動高品質的使用者體驗的重要部分是，確保受管理的用戶端執行的是最新版本的商務用 Skype，除了可確保支援的音訊、影片、網路及 USB 驅動程式都是最新版本。 這提供數個優點，其中包括： 

-   更容易管理幾個版本，與許多版本相比。
-   它提供經驗的一致性等級。
-   它可讓您更輕鬆地排查通話品質和可用性問題。
-   Microsoft 會持續進行整個產品的一般改善及優化。 確保使用者收到這些更新，會降低遇到已解決問題的風險。

將您的部署限制在超過六個月之前的用戶端版本，將可減少所需支援的版本數，改善使用者的整體體驗並改善易管理性。

如果您使用的是 Office 隨選即用，您會自動在六個月內的視窗中。 不需要進一步的動作。

如果您有混合式隨選即用與安裝程式套件 (的 MSI) ，您可以使用報告來確認 MSI 用戶端定期更新。 如果您發現用戶端落後，請與負責管理 Office 更新的小組合作，並確定他們要定期核准和部署用戶端修補程式。

您也必須考慮並確保網路、影片、USB 和音訊驅動程式也有修補。 您可以輕鬆地忽略這些驅動程式，而不要將它們包含在您的修補程式管理原則中。

您可以透過下列連結找到商務用 Skype 的版本號碼：

-   [Microsoft 365 App 之更新的發行資訊](https://docs.microsoft.com/officeupdates/release-notes-office365-proplus)
-   [適用于企業的 Microsoft 365 應用程式更新歷程記錄](https://docs.microsoft.com/officeupdates/update-history-office365-proplus-by-date)
-   [商務用 Skype 的下載及更新](/SkypeForBusiness/software-updates)

### <a name="devices"></a>裝置

若要使用麥克風裝置報表，我們需要瞭解 (MOS) 的平均觀念分數概念。 MOS 是用來估量所察覺之音訊品質的黃金標準測量。 它是從0到5的整數評分來表示。

語音品質的所有測量單位都是人員如何感覺語音品質。 因為它受到人類的認知影響，所以本身就是主觀的。 主觀測試有數種不同的方法。 大多數語音品質測量都是以絕對分類分級為基礎， (ACR) 縮放。

在 ACR 主觀測試中，以統計為單位，大量的使用者在 ()  (不正確的) 中，以較高的速度來評定其體驗品質。 分數的平均值是 MOS。 產生的 MOS 取決於向群組所公開的經驗範圍，以及評級的經驗類型。

因為對即時通訊系統進行語音品質的主觀測試是不切實際的，所以 Microsoft 團隊和商務用 Skype 會使用高級演算法來產生 MOS 值，以客觀預測主觀測試的結果。

可用的一組 MOS 及相關的度量單位可提供透過音訊裝置傳送給使用者的體驗品質。 

您可以為擁有針對小組和商務用 Skype 認證的裝置提供使用者，以減少因裝置本身而遇到負面體驗的可能性 (如此一來，例如，內建的膝上型電腦喇叭和麥克風) 。 如需詳細資訊，請參閱[認證計畫](/SkypeForBusiness/certification/overview)和[合作夥伴解決方案目錄](https://partnersolutions.skypeforbusiness.com/solutionscatalog/personal-peripherals-pcs)中的相關文章。

裝置報告是用來依音量和 MOS 分數來評估裝置使用量 (只) 音訊]，而且可以在 [用戶端 & 裝置] 下的隨附範本中找到。 

> [!IMPORTANT]
> 除非您排除聯盟參與者資料，否則這些報告將包含來自聯盟端點的用戶端遙測。 若要排除同盟端點，您必須在您組織的[租使用者識別碼](CQD-data-and-reports.md#how-to-find-your-tenant-id)中，新增**第二個租使用者 id**的查詢篩選器。 或者，您也可以使用[URL 篩選](CQD-data-and-reports.md#url-filters)來排除同盟參與者遙測。


> [!Note]
> 您可能會注意到，當您查看此報告時，您會看到多次報告相同的裝置。 這是由於將裝置報告給 CQD 的方式所導致。 硬體與作業系統區域設定的差異會導致報告裝置資料的方式差異。

##### <a name="remediation"></a>修正

通常，您需要探索並階段找出未認證的裝置，並將它們取代為認證的裝置。 查看裝置報告時的一些考慮事項包括：

-   適用于團隊和商務用 Skype 的裝置是否已獲認證？ 
-   您可以使用[每個使用者的呼叫分析](use-call-analytics-to-troubleshoot-poor-call-quality.md)來識別特定裝置的使用者。 請檢查以確定他們擁有最新的裝置驅動程式，且其裝置未透過 USB 集線器或塢站連接。 
-   各種不同的驅動程式版本在使用中有多少不同？ 他們是定期進行修補嗎？ 確保定期修補音訊、影片和 Wi-fi 驅動程式，以協助將這些問題排除在品質問題的來源，並讓使用者體驗更具預見性和一致性。

##### <a name="audio"></a>音訊

下一個工作是判斷[認證音訊裝置](https://partnersolutions.skypeforbusiness.com/solutionscatalog/personal-peripherals-pcs)的整體使用量。 我們建議您至少80% 的所有音訊資料流程都使用經過認證的音訊裝置。 最佳做法是將麥克風裝置報告匯出至 Excel，以計算認證或核准裝置的使用方式。 組織通常會保留所有核准裝置的清單，因此篩選及排序資料應該相當簡單。

##### <a name="video"></a>影片

影片驅動程式也很重要，就是繼續更新。 確保經常修正的視訊卡，將無法排除視頻驅動程式，因為這是視頻串流品質較差的來源。 使用[經過驗證的影片裝置](https://partnersolutions.skypeforbusiness.com/solutionscatalog/personal-peripherals-pcs)將有助於確保使用者體驗順暢且高品質。 支援 H-p 原生編碼的視頻裝置是可取的，以減少在視訊會議期間的 CPU 使用量。

##### <a name="wi-fi"></a>Wi-fi

Wi-fi 驅動程式也必須在一般節奏上進行修補，而且應該包含在您的修補程式管理原則中。 您可以透過維護最新的 Wi-fi 驅動程式來修正許多品質問題。 如需優化 Wi-fi 基礎結構的詳細資訊，請參閱[這篇有關 wi-fi 規劃的文章](/skypeforbusiness/certification/networking-wifi)。
